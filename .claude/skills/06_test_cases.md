# Skill 06: 테스트 케이스 도출

## 목적
요구사항(REQ)과 엣지케이스(EC)가 "실제로 동작하는지 어떻게 확인할 것인가"를 정의한다.
TC는 새로 발명하는 것이 아니라, **이미 정의한 REQ·EC를 검증 가능한 형태로 번역**하는 작업이다.
이 단계가 없으면 MUST 기능이 테스트 없이 PRD에 실리고, 추적표의 마지막 칸(TC)이 비어 버린다.

> 선행 조건: `04_edge_cases.md`까지 완료(REQ·EC 확정). 이 스킬은 `05_document_writing.md`로 PRD를 조립하기 **직전**에 수행한다.

---

## 핵심 원칙

1. **TC는 창작이 아니라 번역이다** — REQ의 정상 동작과 EC의 예외 동작을 그대로 "전제–행동–기대결과"로 옮긴다.
2. **기대결과(Then)는 반드시 검증 가능해야 한다** — 관찰 가능한 결과 또는 수치로 끝낸다. "잘 동작한다" ❌ → "3초 이내 응답 시작" ✅
3. **모든 TC는 자신이 검증하는 REQ/EC를 헤딩에 명시한다** — 이 연결이 있어야 추적성 검증 스크립트가 커버리지를 측정한다.
4. **우선순위는 검증 대상에서 상속한다** — MUST REQ를 검증하는 TC는 MUST.

## 실행 프로토콜

### Step 1: 커버리지 목표 설정

PRD에서 아래 목록을 먼저 뽑는다(체크리스트로 쓴다).

- MUST REQ 전체 → 각각 **정상 TC 최소 1개** 필요
- MUST·SHOULD EC 전체 → 각각 **예외 TC 최소 1개** 필요
- NFR 중 측정값이 있는 항목(예: 응답 3초) → **성능 TC 최소 1개**

### Step 2: 정상 TC 도출 (REQ 기반)

각 MUST REQ에 대해 "정상적으로 쓰면 무엇이 보여야 하는가"를 Given–When–Then으로 적는다.

```
REQ-001 (문서 근거 기반 정확 답변)
 → Given: 약관에 명시된 보장 항목 질의
   When: 상담사가 질의 입력
   Then: 문서 근거 답변 + 약관 조항/섹션이 함께 표시된다
```

### Step 3: 예외 TC 도출 (EC 기반)

각 MUST·SHOULD EC를 그대로 예외 TC로 옮긴다. EC의 "발생 조건"이 Given, "처리 방법"이 Then이 된다.

```
EC-01 (문서 근거 없는 질의)
 → Given: 제공 문서에 근거가 없는 질문
   When: 질의 입력
   Then: 추정 답변 대신 "확인 필요/고객센터 유도" 안내가 표시된다
```

### Step 4: 화면 상태 TC 연결 (Phase 2 진행 시)

`09_screen_spec`의 화면 상태(빈 상태 / 로딩 / 오류)는 각각 하나의 TC가 된다.
화면 명세서의 상태 테이블에 `TC-###` 칼럼을 추가해, 명세를 쓰는 순간 TC가 함께 생기게 한다.

### Step 5: 커버리지 측정 (필수)

작성 후 반드시 검증 스크립트로 누락을 확인한다.

```bash
python3 .claude/scripts/validate_traceability.py --strict
```

- "테스트(TC) 없는 MUST REQ" ❌ 가 0이 될 때까지 Step 2로 돌아가 보완한다.
- "테스트(TC) 없는 엣지케이스" ⚠️ 는 SHOULD 이상이면 보완, NICE면 미결(❓)로 남길 수 있다.

### Step 6: 추적 연결

- 각 TC 헤딩에 검증 대상 `REQ-###` / `EC-###`를 괄호로 명시한다(스크립트가 이 괄호를 읽는다).
- 화면 명세(SC)·플로우의 해당 항목에 TC-###를 역방향으로 연결한다.

## 산출물 포맷

PRD `## 테스트 케이스 초안 (TC-###)` 섹션에 아래 형식으로 작성한다.

```markdown
### TC-001 (REQ-001, REQ-004)
- Given: [검증을 시작하는 전제 상태]
- When: [사용자/시스템의 행동]
- Then: [관찰 가능하거나 수치화된 기대 결과]
- 우선순위: MUST   ← 검증 대상 REQ에서 상속

### TC-002 (REQ-002, REQ-005, EC-01)
- Given: 제공 문서에 근거가 없는 질문
- When: 질의 입력
- Then: 추정 답변 대신 "확인 필요/고객센터 유도" 안내가 표시된다
- 우선순위: MUST
```

> 헤딩의 괄호 안에는 **검증 대상 REQ/EC만** 적는다(시나리오·ADR 메모는 괄호 밖 또는 본문에). 괄호 안 ID로 커버리지가 자동 계산되기 때문이다.

### (선택) 커버리지 매트릭스

화면 명세서 추적표 또는 PRD에 요약표를 둘 수 있다.

```markdown
| REQ-ID | 우선순위 | 정상 TC | 예외 TC(EC) |
|--------|---------|---------|-------------|
| REQ-001 | MUST | TC-001 | — |
| REQ-002 | MUST | TC-002 | TC-002(EC-01) |
```

## 완성 체크리스트

- 모든 MUST REQ에 정상 TC가 1개 이상 연결되었는가?
- 모든 MUST·SHOULD EC에 예외 TC가 1개 이상 연결되었는가?
- 측정값이 있는 NFR(예: 응답 3초)에 성능 TC가 있는가?
- 모든 Then이 관찰 가능/수치화되어 있는가? (추상어 금지)
- 각 TC 헤딩에 검증 대상 REQ/EC가 괄호로 명시되었는가?
- `validate_traceability.py --strict` 결과에 ❌(TC 없는 MUST REQ)가 0인가?

## TC 패턴 예시 (AICC 도메인)

아래는 AICC 도메인에서 빠지기 쉬운 TC **예시**다 — 도메인이 다르면 그 도메인의 필수 TC로 교체한다:

| 검증 포인트 | 전형적 Given → Then |
|-------------|---------------------|
| 근거 없는 답변 차단 | 문서에 없는 질의 → 추정 금지·확인필요 안내 |
| 컴플라이언스 위반 차단 | 위반 표현 생성 → 차단 후 재생성(위반 0건) |
| 응답 지연 한계 | 정상 질의 → 최초 응답 3초 이내 시작·초과 시 로깅 |
| 동시성(중복 응대) | 두 상담사 동시 점유 → 1명만 성공·나머지 안내 |
| 외부 연동 실패(CTI/STT) | 연동 끊김 → 수동 대체 경로 제공·오류 로깅 |
| 세션 만료 중 저장 | 만료 상태 저장 시도 → 재인증 유도·데이터 보존 |
