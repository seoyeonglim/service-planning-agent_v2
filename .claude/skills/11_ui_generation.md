# Skill 11: UI 생성

## 목적
화면 명세서 + 디자인 방향을 기반으로 실제 동작 샘플 HTML 화면을 만든다.

---

## 시작 전 필수 확인

UI 생성 전 반드시 이 4개 문서를 모두 읽어라.
하나라도 없으면 생성을 시작하지 말 것.

1. Read docs/[프로젝트명]/ui/01_information_architecture.md  → 전체 화면 목록 파악
2. Read docs/[프로젝트명]/ui/02_user_flow.md                 → 화면 간 이동 경로 파악
3. Read docs/[프로젝트명]/ui/03_screen_spec.md               → 각 화면의 데이터/액션/상태 파악
4. Read docs/[프로젝트명]/ui/04_design_direction.md          → 컬러/타이포/컴포넌트 스타일 파악

추가로 PRD의 REQ 레지스트리와의 추적표가 있어야 한다.

## 화면 생성 순서

1. 공통 컴포넌트 먼저
2. 핵심 플로우 화면
3. 보조 화면
4. 예외 화면

각 화면 생성 후 사용자 확인을 받고 다음 화면으로 진행.

## 화면당 생성 절차 (반드시 준수)

본 HTML을 바로 생성하지 말 것. 아래 순서를 화면마다 반복한다.
모든 시각 산출물은 외부 이미지 생성 API 없이 HTML/Mermaid로 작성한다.

```
① Mermaid 구조도                          → 사용자 확인
② 저해상도 HTML 와이어프레임 (그레이스케일)   → 사용자 확인
③ 주석 HTML 도해 (콜아웃으로 동작 규칙 표기)  → 사용자 확인
④ 본 HTML + Tailwind CSS                    → 사용자 확인
⑤ 다음 화면으로 이동
```

> **진행 모드에 따른 종료 지점:** **Quick(방향 확정·리뷰)** 모드에선 **②(저해상도 와이어프레임)에서 멈추고** 사용자 방향 승인을 받은 뒤 사이클을 끝낸다 — ③④(주석 도해·본 HTML)는 만들지 않는다. **Full** 모드에서만 ③④까지 진행하며, 이때 **④ 본 HTML은 기능명세서(스킬 12) 완료 후**에 생성해 확정된 예외·상태를 반영한다. (CLAUDE.md "진행 모드 선택" 참조)

### ② 저해상도 HTML 와이어프레임 생성 방법

- 그레이스케일(회색 박스/플레이스홀더)만으로 레이아웃·요소 배치를 먼저 확인
- `docs/[프로젝트명]/assets/wireframes/wf-[SC-ID]-[상태명].html` 로 저장
- `.claude/skills/visual_generation/SKILL.md`의 **Type B** 참조

### ③ 주석 HTML 도해 생성 방법

화면 명세서의 **동작 조건·분기·제약·트리거**를, 와이어프레임 위 절대위치 콜아웃(말풍선)으로 표현한다.
`.claude/skills/visual_generation/SKILL.md`의 **Type C** 참조.

- `docs/[프로젝트명]/assets/wireframes/annotated-[SC-ID]-[라벨].html` 로 저장 — **화면별 영구 산출물로 보존**(중간 폐기 금지)
- 도해 파일 상단 주석에 `<!-- 주석도해 SC-## | 명세: ../../ui/03_screen_spec.md (SC-##) | 본 HTML: ../../ui/screens/[파일명] -->` 를 표기해 명세·본 HTML과 상호 링크

**콜아웃 색상 가이드:** 색상별 의미는 `.claude/skills/visual_generation/REFERENCE.md`의 Type C 색상표(정본)를 따른다.

## HTML 파일 구조 규칙

- 기본 템플릿 + Noto Sans KR + 데스크톱 콘솔 기준 레이아웃(좌측 내비 + 본문) 적용
- 각 화면 상단 주석에 **아래 2줄**을 고정 형식으로 표기해, 화면·명세·도해를 SC-ID로 1:1 연결한다(AI·스크립트가 grep으로 따라갈 수 있게):
  ```html
  <!-- SC-## | 화면명 | REQ-###, REQ-### | TC-### -->
  <!-- 명세: ../03_screen_spec.md (SC-##) · 주석도해: ../../assets/wireframes/annotated-SC-##-[라벨].html -->
  ```
  - `[라벨]`은 실제 주석도해 파일의 라벨로 채운다(스킬 11 ③에서 저장한 `annotated-[SC-ID]-[라벨].html`와 동일해야 링크가 유효). 상태 병렬 배치 시 라벨은 `기본`.
  - 1줄차 = 추적 꼬리표(무슨 화면·근거 REQ·검증 TC), 2줄차 = 상호 링크(상세 명세와 시각 도해 위치). **SC-ID가 조인 키**이므로 화면 파일명이 달라도(`sc-01_landing`, `admin_sc-07_sop` 등) SC-##로 매핑된다.
- 본 HTML 생성 후, 화면 명세서의 해당 SC 섹션 `연결 산출물` 줄에 방금 만든 **본 HTML 파일명을 채워** 역링크를 완성한다(양방향 1:1).
- 공통 컴포넌트는 재사용 가능하게 분리(헤더/좌측 내비/카드/버튼)
- 상태 화면(빈/로딩/에러)은 실제 화면에서 전환 가능한 형태로 포함
- 좌측 내비/상단 헤더가 있는 화면은 본문 영역 패딩으로 오버플로우 보정

## 화면 구현 디테일 강제 항목

- 동작 가능한 링크 이동 구현(`onclick` 또는 `<a href>`)
- 입력 유효성 메시지에 실제 텍스트 적용
- 빈 상태/로딩/에러 상태 최소 1개씩 구현
- 스켈레톤/플레이스홀더는 실제 API 지연 시나리오 전제
- 실패 시 액션 복구 버튼(재시도/뒤로가기) 포함

## 더미 데이터 규칙

- 고객명: 김지수, 박서연, 이민수, 최유리, 정하준
- 상담사명: 정해린 상담사, 박도윤 상담사, 김서아 상담사
- 상담관리자명: 한승우 관리자
- 문의 유형: 요금 문의, 배송 지연, 환불 요청, 서비스 장애, 계정 변경
- 채널: 음성콜, 채팅, 카카오 메신저
- 날짜/시각: 오늘 날짜 기준으로 동적 생성
- 영문 사용 금지

## 화면 간 이동 처리

샘플 UI이므로 실제 라우팅 대신 파일 간 링크로 처리:

```html
<button onclick="location.href='agent_02_inquiry_list.html'">상담 목록 보기</button>
<button onclick="history.back()">뒤로</button>
```

## 완성 체크리스트 (화면당)

- 화면 명세와 UI 요소가 1:1 매핑되는가?
- REQ/SC/TC 추적 주석이 포함되는가?
- 모든 액션에 결과/이동/모달 연결이 있는가?
- 빈 상태/로딩/에러 상태 구현이 있는가?
- 한국어 더미 데이터 사용했는가?
- 디자인 규칙(색/타이포/간격/컴포넌트)이 반영되었는가?
- 데스크톱(1280px) 콘솔 기준 레이아웃이 깨지지 않는가?

### 최소 검수용 테스트 시나리오

- TC-### 기준으로 핵심 1개 이상을 화면에서 수동 실행 가능해야 한다.
- 네트워크/에러/권한 제약 같은 실패 경로가 클릭 가능한지 확인한다.
