# Skill 09: 화면 명세서 (Screen Specification)

## 목적
각 화면에 무엇이 있고, 어떤 상태를 가지며, 어떤 행동이 가능한지를 정의한다.
UI 구현자가 추측 없이 그릴 수 있게 만드는 설계도다.

> **진행 모드별 깊이:** **Quick** 모드에선 방향·플로우 확인에 필요한 화면·핵심 항목까지만 경량으로 명세하고 저해상도 와이어프레임으로 넘어간다. **Full** 모드에선 화면당 5항목(데이터·액션·상태·입력 규칙·연결)을 완결한다. (CLAUDE.md "진행 모드 선택" 참조)

---

## 실행 프로토콜

### Step 1: 화면 우선순위 결정

한 번에 전체를 만들지 말고 순차 생성.

1. 사용자 플로우에서 먼저 등장하는 화면
2. 기능 밀집도가 높은 화면
3. 예외 처리가 많은 화면

### Step 2: 화면 항목 정의

각 화면은 필수로 5가지를 작성:

1. **표시 데이터**: 항목 단위로 구체 작성
2. **사용자 액션**: 목적지/결과/제약 포함
3. **상태**: 로딩/빈값/오류/성공
4. **입력 규칙**: 필수/선택, 유효성, 에러 메시지
5. **연결 정보**: REQ-ID/SC-ID/TC-ID

### Step 3: 예외/오류의 미세 명세

성공 플로우만 쓰지 않는다.
- 네트워크 실패 메시지 텍스트
- 동시성 충돌 메시지 텍스트
- 입력 검증 실패 메시지 텍스트
- 권한/세션 만료 처리

### Step 4: 추적성 점검

작성 완료 전 아래를 점검:
- 해당 화면의 `REQ-###` 목록이 PRD에 존재하는가?
- 각 액션이 플로우의 SC-ID 이동 규칙과 일치하는가?
- 각 상태별 문구가 테스트 케이스(TC-###)로 연결되는가?

## 산출물 포맷

```markdown
## 화면 명세서

**작성 기준:** 서비스 구조도 v[X], 사용자 플로우 v[X]
**총 화면 수:** [N개]
**추적 연결:** [REQ-### / SC-ID / TC-ID]

---

### 화면명: [이름] (SC-###)
- **경로:** [URL 또는 화면 식별자]
- **액터:** [이 화면을 사용하는 사용자]
- **진입 경로:** [어디서 오는가]
- **관련 REQ:** [REQ-###]
- **우선순위:** MUST/SHOULD/NICE

#### 표시 데이터
- [데이터 항목 1]
- [데이터 항목 2]

#### 사용자 액션
| 액션 | 이동/결과 | 제약 |
|------|-----------|------|
| [액션 설명] | → [목적지 또는 결과] | [예: 입력 필요/권한 필요] |

#### 입력/유효성
- 필드: [필드명]
- 필수/선택: [필수/선택]
- 형식: [정규식/범위]
- 실패 시 메시지: [문구]

#### 화면 상태
| 상태 | 표시 방법 |
|------|----------|
| [상태명] | [표시 방법/문구] |

#### 레이아웃 힌트
- 상단: [구성 요소]
- 중간: [구성 요소]
- 하단: [구성 요소]

#### 테스트 연결
- TC-###
```

## 완성 체크리스트

- 서비스 구조도의 모든 화면에 명세가 있는가?
- 모든 액션에 목적지(이동/결과)가 명시되었는가?
- 빈 상태/로딩/에러 상태가 모두 정의되었는가?
- 표시 데이터가 구체적인가? ("문의 정보" ❌ → "인입 시각, 채널, 고객명, 문의 유형" ✅)
- REQ/SC/TC 매핑이 모두 존재하는가?
