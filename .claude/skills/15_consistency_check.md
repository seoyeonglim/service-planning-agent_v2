# Skill 15: 내용 정합성 검증

## 목적
추적성 검증(ID 연결)과 기계 검사(consistency_check.py)가 못 잡는 **문장 의미 충돌**을 찾아낸다.
ID는 멀쩡한데 문서끼리 다른 말을 하는 곳 — 수치 불일치(PRD "3초" vs FS "5초"), 조건·분기 상이, 구버전 잔재, 상태 정의 어긋남.

## 실행 시점

- **Phase 1 PRD 완료(baseline) 직전 — PRD 내부 자기 검토(영역 E)**. 초안(v0.x) 매 반복이 아니라 완료 시점에 1회. Quick 모드 초안은 생략 가능.
- Phase 게이트 통과 직전 (특히 Phase 4 완료 후, 개발 핸드오프 전 — 필수)
- 대규모 변경 직후 (CR 여러 건 처리 후, 방향성 변경 반영 후)
- 사용자가 "정합성 검증해줘"라고 요청할 때

## 3계층 검증 구조 (순서대로)

```
① 추적성 검증        python3 .claude/scripts/validate_traceability.py --strict   ← ID 연결
② 기계 정합성 검사    python3 .claude/scripts/consistency_check.py --strict       ← 표 파싱 판정
③ 의미 대조 (본 스킬)  에이전트가 문장을 읽고 충돌 후보를 찾음                      ← 이 문서의 절차
```

①②에서 ❌가 나오면 먼저 해소하고 ③으로 진행한다 (구조가 깨진 상태에서 의미 대조는 낭비).

## Step 1: 검증 영역 분할 — 병렬 에이전트 투입

각 영역을 별도 에이전트(Task)에 맡겨 **동시에** 수행한다. 각 에이전트에게 반드시 포함시킬 지시:

| 영역 | 대조 대상 | 중점 검사 |
|---|---|---|
| A | PRD ↔ 기능명세서 | 처리 규칙·입출력·수치 충돌, 결정표 커버리지, 스코프 크립, EC 처리 방법 일치 |
| B | PRD ↔ UI 문서 4종 (+UI 상호) | 화면 목록, 플로우↔액션 표 모순, 수치·조건, 상태 정의, 용어 드리프트 |
| C | 화면명세 ↔ 와이어프레임·화면 HTML | 커버리지, 상태 구현, 표시 데이터 반영, 추적 주석, 링크 유효성 |
| D | FS·PRD ↔ WBS (·SOW) | 게이트 기준↔단계 일치, 의존 순서, 공휴일 달력 검증, 계약 범위 일치 |
| E | **PRD ↔ PRD (내부 자기 검토)** | **자기모순**(같은 REQ·주제를 두 곳이 다르게 서술, 수치 상충) · **중복**(같은 값·규칙을 여러 곳에 재기술) · **산재**(한 REQ 관련 내용이 여러 절에 흩어짐). 판정은 **PRD 내부 서열** |
| F | **전 산출물 (문체 다이어트)** | **경위 서사·자기 해설 문장·내부 메타·값 중복 재기술·근거 태그 층위 위반**(하류 문서의 날짜·합의 태그, 개정 노트, 제외 사유 반복 — 기준은 CLAUDE.md `근거 태그 층위`·스킬 05). 발견은 **삭제/압축 대체문과 함께** 보고 |

**에이전트 공통 지시문에 반드시 포함:**
- "ID 연결 검증은 이미 통과했다. **내용** 검증이다 — 의미가 충돌하는 것만 보고하라. 표현·상세도 차이는 정상."
- 판정 서열: 문서 간은 `PRD > UI 문서 > 기능명세서·화면 HTML > WBS`(상류 우선). **PRD 내부(영역 E)는 `대전제(범위·문제·목표) > 정본(요구사항 레지스트리·기능 상세) > 참조(사용자 스토리·엣지케이스 서술)`** — 상위층이 옳고 하위를 고침. 단 하위(상세)가 실제로 더 맞으면 하위를 지우지 말고 **상위를 CR로 승격**.
- 발견 형식: 위치(문서·섹션·ID) / 양쪽 원문 짧은 인용 / 어긋남 한 문장 / 심각도(상=의미 충돌·중=수치 불일치·하=표기) / 서열 기준 권장 수정
- "정합 확인된 영역도 마지막에 요약하라" (통과 근거를 남기기 위함)

## Step 2: 발견 통합·분류

4개 결과를 심각도순으로 통합하고, 각 발견을 처리 성격으로 분류한다:

- **[결정]** 기획 판단이 필요 — 상류(PRD)가 침묵하거나 어느 쪽이 옳은지 서열로 판정 불가
- **[기계적]** 방향이 명확 — 서열대로 하류를 고치거나 구버전 잔재를 제거하면 끝
- **[상류 CR]** PRD 자체의 잔재/모순 — v0.x면 직접 수정, v1.0 이후면 CR 절차(스킬 14)

## Step 3: 사용자 확인 — 반드시 물어가며 진행

- **[결정] 항목은 건별로 `AskUserQuestion`** — 선택지 2~4개 + 각각의 트레이드오프. 임의 결정 금지.
- **[기계적] 항목은 묶어서 승인** — "N건을 서열 기준으로 일괄 수정할까요?" 목록 제시 후 진행.
- 결정 결과가 방향성 변경이면 ADR로 기록하고, v1.0 이후면 CR도 남긴다(스킬 14 연계).

## Step 4: 수정 반영 — 파일 영역별 병렬 에이전트

수정도 영역별(파일 소유권 분리)로 병렬 투입한다. 각 수정 에이전트 지시에 **확정된 결정 문구를 그대로** 포함시켜 문서 간 표현이 갈라지지 않게 한다. 담당 파일 외 수정 금지.

## Step 5: 마감 검증

1. `python3 .claude/scripts/validate_traceability.py --strict` → ❌ 0
2. `python3 .claude/scripts/consistency_check.py --strict` → ❌ 0
3. 수정 요약을 사용자에게 보고 (반영 항목 / 건너뛴 항목·사유 / 남은 미결)

## 금지

- ①② 스크립트 검증을 건너뛰고 의미 대조부터 시작 금지
- [결정] 항목을 사용자 확인 없이 임의 판정 금지
- 발견 없이 "정합함"으로 보고할 때 검사한 영역·근거 생략 금지
- 표현 차이·상세도 차이를 충돌로 보고 금지 (의미가 다른 것만)
