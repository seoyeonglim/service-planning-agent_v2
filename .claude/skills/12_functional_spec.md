# Skill 12: 기능명세서 작성 (Functional Specification)

## 목적
PRD(서술)와 화면 명세서(레이아웃) 사이의 빈칸인 **동작 규칙**을 정의한다.
기능명세서는 개발·QA 핸드오프 문서다 — 각 기능의 입력·처리 규칙·출력·예외를
구현하는 사람이 추측 없이 만들 수 있는 단위(`FS-###`)로 쪼갠다.
이 문서가 없으면 개발자는 PRD의 서술을 각자 다르게 해석하고, QA는 검증 기준을 새로 발명하게 된다.

> 선행 조건: Phase 1(PRD 확정) + Phase 2(화면 명세서 `03_screen_spec.md`까지 확정).
> 화면 명세 없이 쓰면 FS의 `연결(SC)` 칸이 비고, 화면 없는 동작 규칙은 검증할 곳이 없다.
> Phase 3(UI 생성)와는 **병행 가능** — 서로 의존하지 않는다.
> **진행 모드:** 이 스킬은 **Full 모드**에서만 수행한다(Quick 모드는 와이어프레임에서 멈추므로 기능명세를 만들지 않음). Full 모드에선 **고해상도 본 HTML(스킬 11 ④) 앞에 기능명세를 완료**하는 것이 기본 순서 — 여기서 새로 드러난 예외·상태가 UI에 반영되도록 한다. (위의 "Phase 3와 병행 가능·무의존" 원칙은 유지되며, 순서만 기능명세를 살짝 앞세워 iterate하는 것이다. CLAUDE.md "진행 모드 선택" 참조)

---

## 핵심 원칙

1. **FS는 창작이 아니라 분해다** — PRD의 REQ 1개를 동작 단위 FS 여러 개로 쪼갠다. PRD에 없는 기능을 FS에서 새로 만들지 않는다(필요하면 PRD에 REQ부터 추가).
2. **모든 FS는 "입력 → 처리 규칙 → 출력 → 예외"로 완결된다** — 4칸 중 하나라도 못 채우면 그 FS는 미결(❓)이다.
3. **모든 FS는 연결(REQ/SC/TC)을 명시한다** — 근거 REQ 없는 FS는 스코프 크립, 화면(SC) 없는 FS는 백엔드 규칙인지 확인, TC 없는 MUST FS는 검증 불가.
4. **한 셀에 안 들어가는 분기는 결정표로 분리한다** — 조건이 3개 이상 얽히면 표의 셀이 아니라 별도 `[결정표 D-#]`로 빼고 셀에서 참조한다.
5. **전 기능에 가로로 걸리는 규칙은 공통 규칙 섹션으로 올린다** — 인증 분기, 보안/마스킹, 폴백 원칙 같은 오버레이를 FS마다 반복해 쓰지 않는다.
6. **기술 구현 방법은 쓰지 않는다** — "무엇이 일어나야 하는가"(규칙)까지만. 어떤 라이브러리/아키텍처로 만들지는 개발 몫.

## 실행 프로토콜

### Step 0: 범위·독자 확인 질문 (AskUserQuestion)

작성 전 반드시 확인한다:

- **명세 범위:** MUST만 / MUST+SHOULD / 전체(NICE·조건부 포함) 중 무엇인가?
- **독자:** 내부 개발팀용인가, 외주·고객사 전달용인가? (전달용이면 전제·책임 구분을 더 명시적으로)
- 스코프에서 제외되는 REQ가 있으면 문서 머리에 "명세 대상 아님"으로 명시한다.

### Step 1: 분해 대상 목록 만들기

PRD의 `요구사항 레지스트리(REQ-###)`를 그대로 가져와 체크리스트로 쓴다.

- MUST REQ 전체 → 각각 **FS 최소 1개** 필요 (커버리지 100%)
- SHOULD·조건부 REQ → 범위 질문(Step 0) 결과에 따라 포함
- REQ를 도메인 그룹(`D#`)으로 묶는다 — 화면 명세의 화면 그룹 또는 사용자 여정 단계를 기준으로

### Step 2: REQ → FS 분해

각 REQ를 "동작 1개 = FS 1행"으로 쪼갠다. 분해 기준:

- 입력이 다르면 다른 FS (예: 신규 입력 수집 vs 기존 값 되묻기)
- 정상 경로와 폴백 경로가 다르면 다른 FS (예: API 연동 계산 vs 예시값 폴백)
- 담당 주체가 다르면 다른 FS (예: 우리가 구현 vs 고객사 후속 — *(전제)* 표기)

```
REQ-050 (정확한 보험료 계산)
 → FS-050: 정확 계산(API 연동) — 입력: 슬롯+API / 출력: 보험료 / 예외: EC-11
 → FS-051: 폴백 계산(예시 요율) — 입력: 슬롯+예시요율 / 출력: 예상 보험료(폴백 표기) / 예외: EC-12
```

### Step 3: 공통 규칙 분리

FS를 쓰다가 같은 문장을 세 번째 반복하고 있다면 멈추고 `§공통 규칙` 섹션으로 올린다.
공통 규칙 표에는 "어느 FS가 이 규칙을 구현하는가"(구현 FS)를 함께 적는다.

전형적 후보: 인증/비인증 분기, 개인정보 마스킹, 컴플라이언스 가드, 세션 유지, 외부 연동 폴백 원칙.

### Step 4: 결정표 분리

셀 안에서 "만약 A면 X, B면 Y, 단 C일 때는…"이 시작되면 결정표로 뺀다.

```markdown
### [결정표 D-1] Intent 라우팅
| 인텐트 | 신호(예) | 분기 동작 | 관련 FS |
|---|---|---|---|
| 상품 문의 | "보장 내용이 뭐야" | RAG 응대(FS-030) | FS-021 |
| 가입 의사 | "가입할래요" | 전환 유도(FS-060) | FS-021 |
```

### Step 5: 추적 연결

- 각 FS 행의 `연결` 열에 `REQ-### / SC-## / TC-###`를 명시한다.
- `예외` 열에는 해당되는 `EC-##`를 명시한다 (PRD 엣지케이스와 연결).
- 우선순위(MUST/SHOULD/NICE)는 근거 REQ에서 **상속**한다 — FS에서 임의로 올리거나 내리지 않는다.

### Step 6: 커버리지 역점검

- 역추적표(§끝)를 만들어 **REQ → FS 방향**으로 누락을 확인한다: FS가 하나도 없는 MUST REQ = ❌.
- `python3 .claude/scripts/validate_traceability.py --strict` 실행 — 기능명세서 자체는 검사 대상이 아니지만, FS 작성 중 발견한 PRD 수정(REQ 추가·정정)이 정합성을 깨지 않았는지 확인한다.

## 산출물 포맷

`docs/[프로젝트명]/fnspec/기능명세서.md` 에 아래 구조로 작성한다.

```markdown
# 기능 명세서 — [프로젝트명]

## 문서 정보
- 문서 버전: v0.x (초안 — 변경 추적은 git 커밋)
- 근거 문서: PRD (`../prd/PRD.md`) · 화면 명세서 (`../ui/03_screen_spec.md`)
- 목적/독자: 개발·QA 핸드오프
- 범위: [포함 우선순위] / 제외: [명세 대상 아닌 REQ와 사유]

## 1. 읽는 법
- 단위: 동작 1개 = FS-### 1행. REQ 1개가 여러 FS로 분해된다.
- 추적: 각 행의 `연결` 열에 REQ / SC / TC 표기.

## 2. 공통 규칙 (전 기능 공통 오버레이·전제)
| 공통 규칙 | 내용 | 구현 FS |
|---|---|---|

## 3. 기능 명세
### D1. [도메인 그룹명]
| FS-ID | 기능명 | 기능 설명·처리 규칙 | 입력 | 출력 | 예외(EC) | 우선순위 | 연결 |
|---|---|---|---|---|---|---|---|

## 4. 결정표
### [결정표 D-1] ...

## 5. 역추적표 (REQ → FS)
| REQ-ID | 우선순위 | FS | 비고 |
|---|---|---|---|
```

> FS-ID 번호는 도메인 그룹별로 십의 자리를 나눠 두면(D1=FS-00x, D2=FS-01x …) 나중에 그룹 내 추가가 쉽다.

## 완성 체크리스트

- 모든 MUST REQ에 FS가 1개 이상 연결되었는가? (역추적표 기준 누락 0)
- 모든 FS의 입력·처리 규칙·출력·예외 4칸이 채워졌거나 미결(❓)로 명시되었는가?
- 모든 FS의 `연결` 열에 REQ가 있는가? (근거 없는 FS = 스코프 크립 → PRD부터 수정)
- 우선순위가 근거 REQ에서 상속되었는가?
- 반복되는 규칙이 공통 규칙 섹션으로 분리되었는가?
- 3개 이상 조건 분기가 결정표로 분리되었는가?
- 외부 제공물·타 주체 책임이 걸린 FS에 *(전제)* / *(조건부)* 가 표기되었는가?
- 기술 구현 방법(라이브러리·아키텍처)이 섞여 있지 않은가?
- 추상어("빠르게", "적절히")가 수치·조건으로 대체되었는가?
