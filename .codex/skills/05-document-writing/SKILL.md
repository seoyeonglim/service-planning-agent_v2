# Skill 05: 문서 작성 노하우

## 목적
읽는 사람이 질문 없이 이해하고 행동할 수 있는 문서를 쓰는 능력.
좋은 기획 문서는 회의 시간을 줄이고 개발 오류를 막는다.

---

## 문서 시작 전 3가지 확인

1. **이 문서의 독자는 누구인가?** (개발자? 대표? 디자이너?)
2. **독자가 이 문서를 읽고 해야 할 행동은 무엇인가?**
3. **결론을 첫 문단에 쓸 수 있는가?**

## 문서 전략

### 기본 원칙
- 결론 우선(Pyramid)
- 한 문장 한 의미
- 추상어 삭제, 수치와 조건으로 대체
- 근거 기반 의사결정 기록(ADR)
- 변경 추적이 가능한 ID 중심 구조
- **PRD 내부 단일 출처**: 값(수치·우선순위·규칙)은 **정본 섹션에 1회만** 쓰고 다른 섹션은 참조한다 — 요구사항·우선순위는 `요구사항 레지스트리`, 상세 규칙·수치는 `기능 상세`가 정본. 문제정의·사용자 스토리·엣지케이스는 자기 각도로 **서술**하되 값을 재기술하지 않는다(값 중복 = 나중에 어긋남). 서열·충돌 판정은 CLAUDE.md `PRD 내부 서열`을 따르고, PRD 완료 전 자기 검토는 스킬 15(영역 E)로 수행한다.

## 독자별 문서 전략

| 독자 | 강조할 것 | 생략할 것 | 분량 |
|------|-----------|-----------|------|
| 개발자 | 조건, 예외, 데이터 형식 | 비즈니스 배경 | 상세하게 |
| 대표/클라이언트 | 임팩트, 수치, 일정 | 기술 세부사항 | 1페이지 요약 |
| 디자이너 | 사용자 감정, 플로우, 상태 | 비용, 공수 | 플로우 중심 |

## 변경 기록(CR) 작성 규칙 — v1.0 확정 이후 필수

v1.0 확정(baseline) 이후의 모든 변경은 `docs/[프로젝트명]/prd/CHANGELOG.md`에 CR(Change Record) 1건으로 기록한다.
목적: **대상 / 변경 전 문구(AS-IS) / 변경 후 문구(TO-BE) / 변경 범위 / 사유 / 승인** 6항목이 한 레코드에서 한눈에 보이게 한다(변경 날짜는 CR 헤딩에 기재).

- CR은 실행 기록, ADR은 의사결정 기록이다. 방향성 결정이 동반되면 ADR을 먼저 남기고 CR의 사유 칸에서 링크한다(ADR 1건 : CR N건 가능).
- **AS-IS/TO-BE는 문구를 그대로 인용**한다(요약 금지 — 구조 변경처럼 인용이 불가능한 경우만 전/후 상태를 서술).
- **변경 범위는 3단계로 작성한다:**
  1. `python3 .claude/scripts/impact_scan.py [대상ID] --md` 실행 → 대상 ID가 등장하는 파일 체크리스트가 자동 생성됨
  2. 체크리스트의 각 파일을 열어 수정 필요 여부를 확인하고, 고친 곳은 체크·"영향 없음"인 곳은 사유와 함께 기록
  3. `python3 .claude/scripts/validate_traceability.py`로 유령/고아 오류가 없는지 재확인
- CR 없이 v1.0 이후 문서를 수정하는 것 금지.
- 변경 요청이 자연어로 들어왔을 때의 전체 절차(REQ ID 매핑 확정 → 영향 스캔 → 변경안 승인 → 파일별 컨펌 수정)는 `14_change_request.md`를 따른다. 이 섹션은 그 결과를 CR 레코드로 남기는 형식 규칙이다.

```markdown
### CR-003 | 2026-07-07 | REQ-041 보험료 재계산 조건 변경

| 항목 | 내용 |
|------|------|
| 대상 | REQ-041 (+ 연쇄: SC-05, FS-012, TC-018) |
| AS-IS | "사용자는 설계 화면에서 나이·성별 변경 시 보험료를 재계산할 수 있다" |
| TO-BE | "사용자는 설계 화면에서 나이·성별·납입기간 변경 시 3초 이내 보험료를 재계산할 수 있다" |
| 변경 범위 | PRD §7 기능상세, 화면명세 SC-05 입력규칙, 기능명세 FS-012 처리규칙, 테스트 TC-018 Then절, WBS W04(영향 없음 확인) |
| 사유 | ADR-07 참조 — 고객사 6/23 방향성 변경 미팅에서 납입기간 슬라이더 요구 |
| 승인 | 사용자 확인 2026-07-07 |
```

## Mermaid 플로우차트 작성 규칙

반드시 이 순서로 작성해라:
1. 정상 플로우 (happy path) 먼저
2. 분기점마다 조건 명시
3. 오류/예외 플로우 별도 표시
4. 각 단계에 액터 명시 (고객 / 상담사 / 시스템 / 외부 연동)

## 문서 완성 후 자가 검토 (발행 전 필수)

- 처음 보는 개발자가 질문 없이 구현할 수 있는가?
- 대표가 30초 안에 핵심을 파악할 수 있는가?
- 확정/가정/미결이 구분되어 있는가?
- 스코프 안/밖이 명시되어 있는가?
- 모든 기능이 REQ-###로 식별되는가?
- 모든 기능의 엣지케이스와 테스트가 연결되어 있는가?
- 추상적 표현이 없는가? ("간편한", "빠른" 등)

하나라도 No면 즉시 수정한다.
