# Skill 05: 문서 작성 노하우

## 목적
읽는 사람이 질문 없이 이해하고 행동할 수 있는 문서를 쓰는 능력.
좋은 기획 문서는 회의 시간을 줄이고 개발 오류를 막는다.

---

## 문서 시작 전 3가지 확인

1. **이 문서의 독자는 누구인가?** (개발자? 대표? 디자이너?)
2. **독자가 이 문서를 읽고 해야 할 행동은 무엇인가?**
3. **결론을 첫 문단에 쓸 수 있는가?**

## 문서 전략

### 기본 원칙
- 결론 우선(Pyramid)
- 한 문장 한 의미
- 추상어 삭제, 수치와 조건으로 대체
- 근거 기반 의사결정 기록(ADR)
- 변경 추적이 가능한 ID 중심 구조

## 독자별 문서 전략

| 독자 | 강조할 것 | 생략할 것 | 분량 |
|------|-----------|-----------|------|
| 개발자 | 조건, 예외, 데이터 형식 | 비즈니스 배경 | 상세하게 |
| 대표/클라이언트 | 임팩트, 수치, 일정 | 기술 세부사항 | 1페이지 요약 |
| 디자이너 | 사용자 감정, 플로우, 상태 | 비용, 공수 | 플로우 중심 |

## Mermaid 플로우차트 작성 규칙

반드시 이 순서로 작성해라:
1. 정상 플로우 (happy path) 먼저
2. 분기점마다 조건 명시
3. 오류/예외 플로우 별도 표시
4. 각 단계에 액터 명시 (고객 / 상담사 / 시스템 / 외부 연동)

## 문서 완성 후 자가 검토 (발행 전 필수)

- 처음 보는 개발자가 질문 없이 구현할 수 있는가?
- 대표가 30초 안에 핵심을 파악할 수 있는가?
- 확정/가정/미결이 구분되어 있는가?
- 스코프 안/밖이 명시되어 있는가?
- 모든 기능이 REQ-###로 식별되는가?
- 모든 기능의 엣지케이스와 테스트가 연결되어 있는가?
- 추상적 표현이 없는가? ("간편한", "빠른" 등)

하나라도 No면 즉시 수정한다.
