# Skill 01: 문제 정의력

## 목적
솔루션이 아닌 진짜 문제를 먼저 찾아내는 능력.
사용자가 "~기능 만들어줘"라고 해도, 먼저 왜 그게 필요한지를 파악한다.

---

## 실행 프로토콜

### Step 1: 5 Whys 적용

사용자가 준 아이디어에 "왜?"를 최대 5번 물어라.
4번째 Why에서 진짜 문제가 보이는 경우가 많다.

**예시 적용:**
```
아이디어: "문의 유형 자동 분류 기능 만들고 싶어요"
Why 1: 왜 자동 분류가 필요한가?       → "상담사가 매번 유형을 수기로 골라서"
Why 2: 왜 수기로 고르는가?            → "인입된 문의를 보고 사람이 판단해야 해서"
Why 3: 왜 사람이 판단해야 하는가?      → "채널마다 문의 형식이 달라 기준이 없어서"
Why 4: 왜 기준이 없는가?              → "유형 체계가 통일돼 있지 않아서"
Why 5: 왜 유형 체계가 없는가?         → "채널별로 따로 운영돼 통합 기준을 만든 적이 없어서"
진짜 문제: "통합 유형 체계가 없어 상담사가 채널마다 문의를 수기 분류하느라 처리 시간이 늘어난다"
```

### Step 2: 문제 정의 수집 질문 (반드시 이 순서로)

아이디어를 받으면 반드시 이 질문들을 해라:

1. "지금 이 문제를 어떻게 해결하고 있나요?" → 현재 해결책 파악
2. "이 문제가 얼마나 자주 발생하나요?" → 빈도/규모 파악
3. "이 문제로 인한 구체적인 피해는 무엇인가요?" → 임팩트 수치화
4. "이 문제가 해결되면 어떤 상태가 되길 원하시나요?" → 성공 기준
5. "지금까지의 증거(관련 로그·기록·고객 피드백(VOC)·클레임·처리량 통계 등)는 무엇이 있나요?" → 근거 확보
6. "이 서비스를 **누가 운영·관리**하나요? 이용자가 만든 것(예약·문의·주문 등)을 **누가 받아서 처리**하나요?" → **운영자·어드민 측 식별**(가장 놓치기 쉬운 부분 — 스킬 02 '사용자 유형 전수 식별' 연계)

### Step 3: 문제 정의서 작성 후 확인받기

질문 후 반드시 이 형식으로 정의하고, 사용자에게 "이게 맞나요?" 확인받아라.
사용자가 "맞아요" 하기 전까지 다음 단계로 넘어가지 말 것.

```markdown
## 문제 정의

| 항목 | 내용 |
|------|------|
| 현재 상황 (AS-IS) | |
| 발생 빈도 | |
| 구체적 피해/손실 | |
| 원하는 상태 (TO-BE) | |
| 성공 기준 (측정 가능) | |
| 증거 소스 | |
| 요구사항 ID(예: REQ-001) | |
```

### Step 4: PRD 진입 전 승인 체크

아래 항목이 채워졌을 때만 다음 단계로 이동:

- 문제 진술이 한 문장으로 명확히 요약되는가?
- 피해/임팩트가 최소 1개 수치로 표현되는가?
- TO-BE 상태가 이용자 관점에서 검증 가능한 행동으로 끝나는가?
- 근거 소스가 1개 이상 제시되었는가?

---

## 문제 패턴 예시 (AICC 도메인)

아래는 AICC 도메인 **예시**다. 도메인이 다르면 그 도메인의 `표면 요청 → 진짜 문제`로 바꿔 적용한다(이 표를 그대로 쓰지 말 것):

| 표면적 요청 | 진짜 문제 |
|-------------|-----------|
| "문의 자동 분류 만들어줘" | 상담사가 유형을 수기 분류하느라 처리 시간이 늘어남 |
| "상담 요약 기능 넣어줘" | 후처리(상담 메모 작성) 시간이 길어 응대 대기가 쌓임 |
| "상담 이력 통합 조회" | 채널이 흩어져 있어 같은 고객의 이전 문의를 못 찾음 |
| "상담사 배정 기능" | 문의가 적합한 상담사에게 안 가 재이관·대기가 발생 |
| "대시보드 보여줘" | 어떤 유형/채널이 응대 부하를 만드는지 모름 |
