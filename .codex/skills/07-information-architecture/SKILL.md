# Skill 07: 서비스 구조도 (Information Architecture)

## 목적
PRD의 기능을 화면 구조로 변환해, 화면 간 누락 없이 제품 전체를 설계한다.

---

## 실행 프로토콜

### Step 1: PRD 기반 화면 추출

PRD의 요구사항 레지스트리(`REQ-###`)를 기준으로 화면 단위를 생성한다.

- 하나의 URL(경로) = 1개 화면
- 모달/바텀시트 = 해당 화면의 하위 상태
- 사용자 유형이 다르면 별도 루트(상담사/상담관리자)

**화면 추출 규칙:**
- 화면명은 PRD의 기능명을 그대로 복제하지 말고 사용자 동작 단위로 정제
- 각 화면에 반드시 `REQ-###` 최소 1개 연결
- 화면당 핵심 액션은 최대 5개

### Step 2: 계층 구조 설계

추출한 화면들을 그룹화해라.

1. 사용자 유형 (상담사 / 상담관리자)
2. 도메인 (상담 처리 / 이력 / 분석 / 설정)
3. 깊이 3단계 초과 금지 (탭 > 페이지 > 상세)

### Step 3: 매핑 포맷 작성

화면 단위 문서에 다음 정보를 의무화한다.

- 화면 ID (예: `SC-AGENT-01`)
- 연결 REQ-ID 목록
- 진입점/종료점
- 에러/빈 상태 존재 여부
- 다음 단계 이동 규칙

### Step 4: 사용자 승인

작성 후 반드시 사용자에게 "누락된 화면이 있나요?" 확인받고,
확인 전까지 사용자 플로우 작성으로 넘어가지 않는다.

---

## 산출물 포맷

```markdown
## 서비스 구조도

**서비스명:** [서비스명]
**사용자 유형:** [상담사 / 상담관리자 / 공통]
**총 화면 수:** [N개]

### [사용자 유형 A] 영역

[서비스명]
├── [그룹명]
│   ├── [화면명] (SC-##) - [한줄 설명] - [REQ-###]
│   └── [화면명] (SC-##) - [한줄 설명] - [REQ-###]
└── [그룹명]
    └── [화면명] (SC-##) - [한줄 설명] - [REQ-###]

### 공통 화면
- 로그인/회원가입
- 오류/로딩/빈 상태 화면

### 매핑표
| SC-ID | 화면명 | REQ-ID | 우선순위 | 비고 |
|-------|--------|--------|----------|------|
```

## 범용 AICC 기본 화면 구조 (예시)

기존 예시는 사용하되, 프로젝트 맞춤으로 교체한다.

```
범용 AICC
├── 상담사 워크스페이스 (Agent)
│   ├── 홈 - 실시간 대기열/내 상담 현황 요약
│   ├── 상담 처리
│   │   ├── 인입 문의 목록(채널 통합)
│   │   ├── 상담 상세/응대
│   │   └── 문의 유형 분류
│   ├── 상담 이력
│   │   ├── 이력 목록
│   │   └── 이력 상세/이전 문의 요약
│   └── 설정
│       ├── 상태(응대 가능/이석) 설정
│       └── 알림 설정
└── 상담관리자 콘솔 (Supervisor/QA)
    ├── 대시보드 - 인입량/처리량/SLA
    ├── 모니터링
    │   ├── 상담사별 현황
    │   └── 실시간 상담 모니터링
    └── 품질 관리
        ├── 분류 정확도/오분류 교정
        └── 유형 체계 관리
```
