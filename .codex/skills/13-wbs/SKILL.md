# Skill 13: WBS 작성 (Work Breakdown Structure)

## 목적
확정된 스코프(REQ·FS)를 **일정과 리소스**로 변환한다.
WBS는 두 개의 판으로 구성된다 — ① **기능판**(무엇을 몇 주차에 만드는가, 티켓 생성 기준)
② **리소스판**(누가 몇 주차에 얼마나 투입되는가).
이 문서가 없으면 일정은 "느낌"으로 약속되고, 공휴일·외부 의존이 터진 뒤에야 발견된다.

> 선행 조건: `12_functional_spec.md` 완료(FS-### 확정).
> WBS의 작업 단위는 Epic=REQ / Story=FS 매핑을 쓰므로, FS 없이 쓰면 작업 분해 기준이 없어
> "개발", "테스트" 같은 검증 불가능한 덩어리가 된다.

---

## 핵심 원칙

1. **명목일이 아니라 가용일로 계획한다** — 주 5일이 아니라 "5일 − 공휴일 = 실제 일할 수 있는 날"이 계획 단위다. 실제 달력(주말·법정공휴일·대체공휴일)을 반드시 대조한다.
2. **작업 단위는 발명하지 않는다** — Epic=REQ, Story=FS. 기능명세서에 없는 작업이 필요하면(인프라 셋업 등) 별도 섹션으로 명시하고 근거를 적는다.
3. **외부 의존(전제)은 일정의 지뢰다** — 타 주체 제공물(데이터·API·계정·검토)이 걸린 항목은 🔴 전제로 표기하고, "미제공 시 어느 주차로 후퇴하는가"를 미리 정한다.
4. **대외 약속은 불변, 내부 목표는 선행** — 고객사 보고 게이트는 그대로 두고, 내부 목표를 앞당겨 버퍼를 만든다. 내부 선행이 무너져도 공식 일정은 지켜지는 구조가 핵심이다.
5. **임의 산정값은 임의라고 쓴다** — 킥오프일·투입률 등 가정에 기반한 수치는 "(가정, 착수 시 확정)"을 명시한다. 확정처럼 보이는 가정이 가장 위험하다.

## 실행 프로토콜

### Step 0: 일정 전제 확인 질문 (AskUserQuestion)

작성 전 반드시 확인한다:

- **용도:** 내부 운영용인가, 고객사 제출용인가? (내부용이면 선행 전략·리스크를 솔직하게, 제출용이면 공식 게이트 중심으로)
- **킥오프일과 스프린트 길이:** 킥오프 예정일이 주말/공휴일은 아닌지 실제 달력으로 검증한다.
- **게이트/마일스톤:** 중간 검수 시점과 통과 기준(수치)이 계약·PRD에 이미 있는가?
- **투입 인원:** 멤버(또는 역할)별 투입 가능 기간과 상한 투입률.

### Step 1: 주차 체계 수립 (가용일 계산)

킥오프일부터 오픈까지를 주 단위(`W01~Wnn`)로 편성하고, 주마다 가용일을 계산한다.

- 해당 기간의 **한국 법정공휴일·대체공휴일**을 실제 달력에서 뽑아 반영한다 (임시공휴일은 미반영·발생 시 갱신을 명시).
- 가용일 3~4일인 주는 ⚠️ 표시 — 스프린트 플래닝에서 용량을 그만큼 줄여 잡는다.
- 스프린트 사이 또는 게이트 뒤에 **버퍼 주**를 배치한다 (보완·이월 흡수 전용, 기능 배치 금지).

### Step 2: 단계·게이트 배치

PRD의 추진 단계(P1/P2/…)를 주차 구간에 매핑하고, 각 게이트에 **수치화된 통과 기준**을 명시한다.
"검수"가 아니라 "Intent 정확도 75%·SOP 사인오프"처럼 — 기준 없는 게이트는 통과도 실패도 판정할 수 없다.

### Step 3: 기능판 작성 (Epic=REQ / Story=FS)

FS를 주차(스프린트)에 배치한다.

- 각 행: `FS-ID | 작업명 | 주차 | 담당 | 의존(선행 FS·전제) | 근거(REQ)`
- 의존 관계가 있는 FS는 선행 FS보다 앞 주차에 배치할 수 없다 — 배치 후 역방향으로 검증한다.
- 🔴 전제(외부 제공물) 항목은 "제공 가정 시점"과 "미제공 시 후퇴 규칙"을 함께 적는다.

### Step 4: 리소스판 작성

멤버(역할)별 × 주차별 투입률(%)을 표로 만든다.

- 한 사람의 주차 합계가 100%를 넘으면 배치 오류다.
- 저가용 주(⚠️)에 MUST 기능이 몰려 있으면 재배치한다.

### Step 5: 버퍼·리스크 전략

- **내부 선행 목표:** 각 공식 게이트 대비 내부 목표를 1~2주 앞으로 설정하고, 선행으로 비는 공식 구간의 용도(다음 단계 조기 착수·검수 대응)를 명시한다.
- **요주의 주차:** 저가용 주와 겹치는 핵심 스프린트(연휴와 겹친 통합 테스트 등)를 짚고 대응(일정 이동·선확보)을 적는다.

### Step 6: 미결 관리

확정 못 한 항목(임시공휴일, 투입률 미정, 전제 제공일 미정)은 미결(❓) 목록으로 남긴다.
미결이 있어도 v0.x 초안은 낼 수 있지만, **무엇이 가정인지 문서 안에서 구분되어야 한다.**

## 산출물 포맷

`docs/[프로젝트명]/wbs/WBS_v0.x.md` 에 아래 구조로 작성한다.
(내부용/제출용을 구분해야 하면 파일명에 명시: `WBS_내부_v0.1.md`)

```markdown
# WBS — [프로젝트명]

## 문서 정보
- 문서 버전: v0.x (초안 — 변경 추적은 git 커밋)
- 근거 문서: PRD (`../prd/PRD.md`) · 기능명세서 (`../fnspec/기능명세서.md`)
- 용도: [내부 리소스 계획 / 티켓 생성 기준 / 고객사 제출]
- 일정 전제: 킥오프 YYYY-MM-DD(요일 검증됨) · 주 5일 · [가정임을 명시]
- 가용일 기준: 주말·법정공휴일(대체 포함) 제외. 반영 공휴일: [목록]. 임시공휴일 미반영.

## 1. 주차 체계 (W01~Wnn) — 가용일 반영
| 주차 | 기간 | 가용일 | 공휴일 | 스프린트 | 단계 | 비고 |
|---|---|:---:|---|---|:---:|---|

### 1-1. 내부 선행 운영 (공식 게이트 대비 버퍼)
| 게이트 (통과 기준) | 공식(보고) | 내부 목표 | 확보 버퍼 |
|---|:---:|:---:|:---:|

## 2. 리소스판 — 멤버별 주차 투입률
| 멤버(역할) | W01 | W02 | … | 합계(MM) |
|---|:---:|:---:|:---:|:---:|

## 3. 기능판 — Epic=REQ / Story=FS
### P1. [단계명] (W01~Wnn)
| FS-ID | 작업명 | 주차 | 담당 | 의존·전제 | 근거 |
|---|---|:---:|---|---|---|

## 4. 리스크·요주의 주차
## 5. 미결(❓)
```

## 완성 체크리스트

- 킥오프일·게이트일이 실제 달력으로 검증되었는가? (주말/공휴일에 걸린 날짜 없음)
- 기간 내 법정공휴일·대체공휴일이 주차 표에 반영되었는가?
- 가용일 3~4일 주에 ⚠️가 표시되고 용량 축소가 반영되었는가?
- 모든 MUST FS가 주차에 배치되었는가? (배치 안 된 MUST = ❌)
- 의존 관계가 있는 FS가 선행 FS보다 앞 주차에 배치되어 있지 않은가?
- 🔴 전제 항목마다 제공 가정 시점과 미제공 시 후퇴 규칙이 있는가?
- 각 게이트에 수치화된 통과 기준이 있는가?
- 멤버별 주차 투입률 합계가 100%를 넘는 곳이 없는가?
- 버퍼 주가 있고, 버퍼 주에 기능이 배치되어 있지 않은가?
- 가정값에 "(가정, 착수 시 확정)"이 명시되었는가?
