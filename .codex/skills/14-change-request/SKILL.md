# Skill 14: 변경 요청 처리

## 목적
v1.0 확정(baseline) 이후 자연어로 들어온 요청("보험료 계산에 납입기간도 들어가야 해요")을
**기존 요구사항의 변경(A플로우)인지, 새 요구사항의 추가(B플로우)인지 판별**한 뒤,
영향 파악 → 사용자 승인 → 문서별 컨펌 반영 → 검증·기록의 순서로 안전하게 처리하는 능력.
**삭제(요건이 빠지는 것)도 변경의 일종이다** — "TO-BE가 폐기인 변경"으로 A플로우에서 처리한다.

이 단계가 없으면 에이전트가 어떤 REQ인지 확인도 받지 않고 여러 문서를 한꺼번에 고치거나,
스코프가 늘어나는 추가 요청을 조용히 끼워 넣어 일정·범위 합의가 깨진다.

> 적용 시점: **v1.0 확정 이후의 모든 요청.** 초안(v0.x) 단계의 수정은 이 스킬 대상이 아니다
> (git 커밋으로 추적, CLAUDE.md 버전 생애주기 참조).
> CR 레코드 형식과 작성 규칙은 `05_document_writing.md`의 "변경 기록(CR) 작성 규칙"을 따른다.

---

## 핵심 원칙

1. **매핑/판별 확정 전 수정 금지** — 요청이 어느 REQ의 변경인지, 아니면 신규인지 사용자에게 확인받기 전에는 어떤 문서도 고치지 않는다.
2. **고치기 전에 영향부터** — 변경은 `impact_scan.py`로, 추가는 일정·MVP 경계 재판정으로 "이 요청의 대가"를 먼저 보여준다.
3. **일괄 수정 금지, 파일 단위 컨펌** — 두 플로우 모두 연쇄 문서는 한 파일씩 "이렇게 반영하겠습니다 / 영향 없습니다"를 보여주고 컨펌받은 뒤 진행한다.
4. **변경과 추가는 도구가 다르다** — 변경은 impact_scan이 위치를 찾아주지만, 추가는 새 ID가 아직 어디에도 없어 스캔할 것이 없다. 추가에서는 `validate_traceability.py --strict`가 누락 적발의 주역이다.
5. **추가를 기존 REQ에 억지로 끼워 넣기 금지** — 스코프 확장은 숨기지 말고 ADR로 드러낸다.
6. **삭제도 변경이다 — 단, 물리 삭제는 금지** — 레지스트리 행과 ID는 영구 보존(결번)하고 우선순위 칸만 "폐기"로 바꾼다. 행을 지우면 CHANGELOG·ADR에 남은 과거 언급이 전부 유령 참조 ❌가 되어 검증이 영원히 실패한다.

---

## Step 0 (공통): 접수 및 판별

1. PRD의 요구사항 레지스트리에서 요청 내용과 관련된 REQ 후보를 찾는다 (기능명·키워드·화면 언급 단서 활용).
2. `AskUserQuestion`으로 판별을 확정받는다:
   - 후보가 1개여도 **반드시 확인** — "REQ-041(보험료 재계산) 변경 말씀이시죠?"
   - 후보가 여러 개면 선택지 2~4개로 제시 (각 선택지에 REQ 원문 요약을 description으로)
   - 기존 REQ의 확장으로 볼 수도, 신규로 볼 수도 있는 애매한 경우엔 **양쪽을 선택지로** 제시 — "REQ-041의 조건 변경으로 처리 / 신규 요구사항으로 추가"
3. 분기:
   - **기존 REQ에 매핑 확정 → A플로우(변경)**
   - **매핑되는 REQ 없음 (또는 신규로 확정) → B플로우(추가)**
   - **삭제 요청 → A플로우(삭제형 변경)** — 단, 판별 시 삭제 유형도 함께 확정한다: **완전 폐기 / 다음 버전으로 연기(보류 구획 이동) / 다른 REQ로 흡수** 중 무엇인지. 흡수라면 흡수 대상 REQ의 일반 변경으로 전환한다.
4. 하나의 요청이 여러 REQ에 걸치면 전부 나열해 확정받고, 이후 단계는 확정된 ID 집합 전체로 진행한다.

---

## A플로우: 기존 요구사항 변경

### A1: 영향 범위 스캔 (수정 전에!)

```bash
python3 .claude/scripts/impact_scan.py [확정된 ID들] --md
```

- 출력된 체크리스트로 "이 변경은 문서 N개, M곳에 영향을 줍니다"를 사용자에게 보고한다.
- 이 체크리스트가 A4의 작업 순서표이자 CR `변경 범위` 칸의 초안이 된다.
- 스캔 결과가 비어 있으면(종료코드 1) ID 매핑이 틀렸을 가능성 — Step 0으로 돌아간다.

### A2: AS-IS / TO-BE 변경안 승인

1. PRD에서 해당 REQ의 **현재 문구를 그대로 인용**해 AS-IS로 제시한다.
2. 요청을 반영한 TO-BE 문구를 제안한다 (추상어 금지 — 수치·조건으로).
3. 방향성 결정이 동반되면 **ADR을 먼저 작성**하고 변경안에 링크한다.
4. 사용자가 변경안을 승인하기 전에는 수정을 시작하지 않는다.

### A3: PRD부터 수정

원본 정의가 먼저다. 승인된 TO-BE로 PRD(레지스트리 + 기능 상세 등 PRD 내 등장 위치)를 수정한다.
연쇄 문서는 아직 건드리지 않는다.

### A4: 연쇄 문서를 하나씩 컨펌받으며 수정

A1 체크리스트의 나머지 파일을 **위에서부터 한 파일씩** 처리한다. 파일마다:

1. 해당 ID가 등장하는 부분(줄)을 인용해 보여준다.
2. 판단을 제시한다 — **수정 필요**(어떻게 고칠지) 또는 **영향 없음**(왜 그대로여도 되는지 사유).
3. `AskUserQuestion`으로 컨펌받는다 (예: "수정 진행 / 다르게 수정 / 영향 없음 처리").
4. 컨펌된 대로 수정하고 체크리스트에 결과를 기록한다 — 고친 곳은 체크, 영향 없음은 사유와 함께.

- 같은 파일 안의 여러 등장 위치는 한 번에 보여주되, **컨펌 단위는 파일 1개**다.
- 컨펌 없이 다음 파일로 넘어가거나 여러 파일을 한꺼번에 수정하는 것 금지.

### A5: 검증 + CR 기록

1. `python3 .claude/scripts/validate_traceability.py --strict` → 수정 과정에서 ID가 깨지거나 누락되지 않았는지 확인, ❌ 오류 0이어야 종료 가능.
2. `prd/CHANGELOG.md`에 CR-### 레코드 작성 — `변경 범위` 칸에 A4의 체크리스트 결과를 그대로 옮긴다.
3. 사용자 최종 확인 날짜를 CR `승인` 칸에 기록한다.

### A플로우 변형: 삭제형 변경 (TO-BE = "폐기")

삭제도 골격은 A1~A5 그대로다. 단계별 차이만 지킨다:

- **A2 (변경안)**: TO-BE는 `"폐기 — [사유]"`. **ADR 필수** — 왜 빠지는지, 어떤 조건이면 되살리는지 기록.
- **A3 (PRD)**: 레지스트리 **행 삭제 금지** — 우선순위 칸을 `폐기(CR-###)`로 바꾼다. 폐기된 REQ는 MUST/SHOULD/NICE 어느 것도 아니게 되어 TC 커버리지·고아 검사에서 자동 제외된다(검증 스크립트 수정 불필요). ID는 영구 결번.
- **A4 (연쇄 문서)**: 파일별 판단 기준이 둘로 갈린다 —
  - **활성 문서**(`ui/`, `fnspec/`, `wbs/`): 참조 제거 또는 섹션 삭제
  - **기록 문서**(CHANGELOG, ADR, 변경 이력 표): "영향 없음 — 기록 보존" 처리 (언급 제거 금지)
  - 이 REQ만 검증하던 TC, 이 REQ만 쓰던 EC·화면(SC)·FS는 **같이 폐기 대상**으로 목록에 올려 컨펌받는다.
- **A5 (검증·CR)**: 폐기 REQ는 정의가 레지스트리에 남아 있어 활성 문서의 잔존 참조를 스크립트가 못 잡는다 — **A1 체크리스트를 끝까지 완주하는 것이 곧 게이트**다. CR의 TO-BE 칸은 "폐기". WBS에서 일정 여유가 생기면 함께 보고한다.

## B플로우: 요건 추가 (신규 REQ)

### B1: 스코프 판정 — "이 추가의 대가"를 먼저 합의

1. **ID 채번**: 레지스트리 마지막 번호 + 1 (결번 재사용 금지 — 과거 CR·ADR이 참조할 수 있다).
2. **우선순위 제안**: MUST/SHOULD/NICE — `03_structuring.md` 기준 적용. 추가로 MUST 비율이 40%를 넘게 되면 과설계 경고와 함께 보고한다.
3. **포함 시점 판정**: 이번 버전(v1.x)에 포함할지, v2 백로그로 미룰지 — MVP 경계 기준(`03_structuring.md`)으로 판단 근거를 제시하고 사용자가 결정한다. WBS가 이미 있으면 일정 영향(어느 주차가 밀리는지)을 함께 보고한다.
4. 스코프 확장 결정은 **ADR로 기록**한다 (포함/보류 어느 쪽이든 — 버려진 선택지와 이유 포함).

### B2: 정의 작성 — 신규 REQ를 "완전한 요구사항"으로

Phase 1 스킬을 축약 적용해 새 REQ를 기존 REQ와 같은 완성도로 만든다:

1. **문제 근거** (`01` 축약): 왜 필요한가 — 현재 어떻게 해결 중이고, 어떤 피해/가치가 있는지 최소 1개 수치 확보
2. **엣지케이스** (`04` 축약): 22문항 체크리스트를 훑어 EC 최소 1개 도출·채번
3. **테스트 케이스** (`06` 축약): MUST면 정상 TC 필수 + EC마다 예외 TC — `### TC-### (REQ-###, EC-##)` 괄호 연결 준수
4. PRD의 요구사항 레지스트리·기능 상세·EC·TC 섹션에 등록 → 사용자 확인

### B3: 연쇄 문서에 전파 — 산출물 단계 체크리스트로

새 ID는 아직 어디에도 없으므로 **impact_scan은 쓸 수 없다.** 대신 프로젝트에 존재하는 산출물 단계를 위에서부터 훑는다 (없는 단계는 건너뜀). 파일 단위 컨펌 방식은 A4와 동일:

| 순서 | 문서 | 판단할 것 |
|---|---|---|
| 1 | IA (`ui/01`) | 기존 화면(SC)에 얹는가, 신규 SC를 채번하는가 |
| 2 | 사용자 플로우 (`ui/02`) | 기존 플로우에 경로 추가인가, 신규 플로우인가 (예외 경로 포함) |
| 3 | 화면 명세 (`ui/03`) | 해당 SC 명세에 표시 데이터·액션·상태 추가. 신규 화면이면 Phase 3 절차(스킬 11)로 HTML까지 |
| 4 | 기능명세서 (`fnspec/`) | REQ → FS-### 분해 (`12` 기준: 입력→처리→출력→예외 4칸 완결) |
| 5 | WBS (`wbs/`) | 주차 배치 — B1에서 합의한 일정 영향을 반영 |

### B4: 검증 — validate가 누락을 적발한다

```bash
python3 .claude/scripts/validate_traceability.py --strict
```

새 MUST REQ를 등록하는 순간 "TC 없는 MUST REQ ❌", "화면에 없는 MUST REQ(고아) ❌"가
빠진 전파를 자동으로 잡아준다. **❌ 오류 0이 될 때까지 B2~B3을 반복**한다.

### B5: CR 기록

- **대상**: 신규 REQ-### + 함께 채번된 연쇄 ID(EC/TC/SC/FS)
- **AS-IS**: "없음 (신규 추가)"
- **TO-BE**: 새 요구사항 문구
- **변경 범위**: B3 체크리스트 결과 (추가된 문서·섹션 목록)
- **사유**: B1의 ADR 링크
- **승인**: 사용자 확인 날짜

## 게이트 (하나라도 어기면 진행 중단)

- [ ] 변경인지 추가인지 사용자에게 판별받았는가? (Step 0)
- [ ] **A**: 수정 **전에** impact_scan 실행 + AS-IS/TO-BE 승인을 받았는가? (A1~A2)
- [ ] **B**: 채번·우선순위·포함 시점을 승인받고 ADR을 남겼는가? (B1)
- [ ] **B**: 새 REQ에 EC·TC가 연결되었는가? (B2)
- [ ] **삭제**: 레지스트리 행을 폐기 표기로 남겼는가(물리 삭제 안 함) + 전용 연쇄 산출물(TC/EC/SC/FS)의 동반 폐기를 컨펌받았는가?
- [ ] 연쇄 문서를 파일 단위로 컨펌받으며 반영했는가? (A4 / B3)
- [ ] `--strict` 검증 통과 + CR 기록을 남겼는가? (A5 / B5)

## 금지사항

- 판별(매핑/신규) 확정 전 문서 수정 금지
- (A) impact_scan 실행 전 연쇄 문서 수정 금지
- (B) ADR 없이 스코프 확장 금지, 결번 ID 재사용 금지, EC·TC 없는 신규 REQ 등록 금지
- (삭제) 레지스트리 행 물리 삭제 금지, 폐기 ID 재사용 금지, 기록 문서(CHANGELOG·ADR)에서 폐기 REQ 언급 제거 금지
- 여러 파일 일괄 수정 금지 (파일 단위 컨펌 필수)
- CR 없이 v1.0 이후 문서 수정 금지
- 추가 요청을 기존 REQ에 억지로 끼워 넣어 스코프 확장을 숨기는 것 금지
