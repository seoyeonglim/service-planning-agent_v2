# Skill 04: 엣지케이스 감지

## 목적
정상 플로우의 구멍을 미리 찾아내는 능력.
개발 후 발견하면 비용이 10배, 서비스 중 발견하면 100배 든다.

---

## 실행 프로토콜

### Step 1: 우선순위 분류

모든 엣지케이스를 아래 기준으로 분류한다.

- 영향도: 매출 손실 / 사용자 불신 / 운영 리스크
- 빈도: 높음 / 중간 / 낮음
- 대응 난이도: 쉬움 / 보통 / 어려움
- 추천 분류: MUST / SHOULD / NICE

### Step 2: 엣지케이스 체크리스트 (6개 카테고리)

아래 **6개 카테고리(동시성·네트워크/시스템·데이터 경계·시간/날짜·상태 변화·권한/접근)는 모든 도메인 공통**이므로 각 기능에 반드시 전부 적용한다. 다만 **아래 질문 문장은 AICC 예시**이니, 그 도메인의 언어로 치환해서 묻는다 — 예: "두 상담사가 같은 문의를 동시에 잡으면" → 예약앱이면 "두 손님이 같은 시간대를 동시에 예약하면?".

### 동시성
- [ ] 두 상담사가 같은 문의를 동시에 잡으면(중복 응대)?
- [ ] 같은 사용자가 저장 버튼을 2번 연속 누르면?
- [ ] 관리자가 유형 체계를 수정하는 동시에 상담사가 그 유형으로 분류하면?

### 네트워크/시스템
- [ ] 상담 저장 직전에 시스템이 종료되면?
- [ ] 분류 결과 전송 중 서버가 다운되면?
- [ ] 채널 연동(콜/챗/메신저)이 일시적으로 끊기면?

### 데이터 경계값
- [ ] 문의 내용에 특수문자/이모지가 포함되면?
- [ ] 전화번호/고객 식별자를 잘못된 형식으로 받으면?
- [ ] 상담 메모 최대 글자 수 초과 입력 시?

### 시간/날짜
- [ ] 자정을 넘겨 이어지는 상담의 처리 시간 집계는?
- [ ] 영업시간 외(야간)에 인입된 문의 처리는?
- [ ] 응대 SLA가 1분 남았을 때 처리는?
- [ ] 이미 종료된 상담을 다시 열려고 하면?

### 상태 변화
- [ ] 상담 진행 중 담당 상담사가 로그아웃/퇴사하면?
- [ ] 사용 중이던 유형 코드가 삭제되면?
- [ ] 고객이 상담 도중 채널을 바꾸면(콜→챗)?
- [ ] 관리자가 배정 규칙을 바꿨는데 진행 중인 상담이 있으면?

### 권한/접근
- [ ] 세션 만료 상태에서 상담 저장 버튼을 누르면?
- [ ] 상담사가 다른 팀의 상담 이력을 조회하려 하면?
- [ ] 탈퇴/삭제된 고객의 상담 데이터는 어떻게 처리되는가?

### Step 3: 문서 포맷 및 추적

발견된 엣지케이스는 반드시 이 형식으로 기록해라:

```markdown
### EC-[번호]: [상황명] (REQ-###)

| 항목 | 내용 |
|------|------|
| 발생 조건 | |
| 예상 빈도 | 높음 / 중간 / 낮음 |
| 사용자 영향 | |
| 처리 방법 | 무시 / 안내 메시지 / 자동 복구 / 강제 중단 |
| 대응 우선순위 | MUST / SHOULD / NICE |
| 처리 화면/플로우 | |
| MVP 포함 여부 | 필수 / v2 / 보류 |
```

### Step 4: 구현 전 승인

- MUST 엣지케이스가 하나라도 누락되면 UI 문서 작성 단계로 이동 금지
- MUST/SHOULD는 PRD의 NFR 및 테스트 케이스와 연결하여 추적성 유지

## 엣지케이스 예시 (AICC 도메인)

아래는 AICC 도메인에서 자주 나오는 **예시**다 — 도메인이 다르면 그 도메인의 시나리오로 교체한다:

| 시나리오 | 발생 빈도 | 우선순위 |
|----------|-----------|----------|
| 중복 응대 (두 상담사가 같은 문의) | 높음 | MUST |
| 분류 불가/모호한 문의 | 높음 | MUST |
| 상담 중 담당자 이탈(이관 필요) | 중간 | MUST |
| 채널 전환 (콜→챗→메신저) | 중간 | SHOULD |
| 상담 저장 실패 | 중간 | SHOULD |
| 피크 시간 대량 인입(대기열 폭주) | 낮음 | NICE |
| 외국어 문의 (언어 자동 감지 필요) | 낮음 | NICE |
