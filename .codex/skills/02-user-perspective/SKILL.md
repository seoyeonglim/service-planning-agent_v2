# Skill 02: 사용자 관점 전환

## 목적
내 관점이 아닌 실제 사용자의 맥락에서 생각하는 능력.
기획의 모든 결정은 "이 사람이라면?"으로 검증한다.

---

## 사용자 파악 질문 (기획 시작 전 필수)

- 주요 사용자는 몇 살대인가? (디지털 친숙도가 달라짐)
- 이 기능을 언제, 어디서 사용하는가? (동시 응대 중? 모니터링 중?)
- 업무 시스템 사용 숙련도는? (단축키를 쓰는가, 마우스 위주인가?)
- 이 서비스 사용 직전에 어떤 행동을 하는가?
- 이 서비스 사용 직후에 무엇을 하는가?
- 가장 두려워하는 실패 시나리오는?
- 사용자가 현재 대체하고 있는 기존 과정은 무엇인가?

## 사용자 유형 전수 식별 (이용자 + 운영자, 필수)

서비스에는 보통 **두 축**의 사용자가 있다. 한쪽만 보면 반드시 빠진다 — 특히 **운영자(어드민)**를 놓치기 쉽다.

1. **이용자(end-user)** — 서비스를 소비하는 사람. 예: 예약하는 손님, 문의하는 고객, 주문하는 구매자.
2. **운영자·관리자(operator/admin)** — 서비스를 **운영·관리**하고, 이용자가 만들어낸 것(예약·문의·주문 등)을 **받아서 처리**하는 사람. 예: 예약을 받는 사장/점주/스태프, 문의를 응대하는 상담사, 지표를 보는 운영팀.

**규칙:**

- 어떤 서비스를 기획하든 **두 축을 모두 식별**하고 각각 페르소나·핵심 니즈를 정의한다. 위의 `사용자 파악 질문`은 **각 축마다** 적용한다.
- **이용자만 정의하고 운영자를 빠뜨리면 미완성**이다. (예: "예약 앱"인데 그 예약을 받아 관리할 **사장용 화면이 없음** → 서비스가 성립하지 않는다.)
- 운영자 측은 대개 **관리·모니터링 화면(어드민/백오피스)**을 동반한다 — 이용자가 생성한 데이터를 조회·처리·설정하는 곳. 이 축을 정의하면 Phase 2 IA에서 어드민 화면이 자연히 도출된다.
- 운영자 측이 진짜 없는 단방향 서비스(예: 순수 계산기·단순 정보 조회)라면, **"운영자 없음 + 이유"를 명시적으로 기록**한다. 암묵적 누락 금지.

> 두 축을 확정하기 전에는 다음 단계로 넘어가지 않는다.

## 사용성 번역 규칙

요청한 기능을 사용자 언어로 변환할 때 아래 형식을 기본으로 제시한다.

- 목적: 어떤 일을 성취하려 하는가?
- 빈도: 하루/주간 예상 횟수
- 우려: 실패시 손실(시간/리스크/감정)
- 허용 실패: 어느 정도까지 허용되는가?

### 사용자 언어 번역 규칙

기획 문서에서 절대 쓰지 말아야 할 표현과 대체어:

| ❌ 금지 표현 | ✅ 대체 표현 |
|-------------|-------------|
| "간편하게" | "3번 탭 이내로" |
| "빠르게" | "2초 이내에" |
| "쉽게 찾을 수 있는" | "메인 화면에서 1단계로 접근 가능한" |
| "알림을 받는다" | "신규 문의 인입 시 3초 이내 화면 알림을 수신한다" |
| "편리한 UI" | (기획 문서에서 UI 품질 표현 금지) |
| "최적화된" | (구체적 조건으로 대체) |
| "직관적인" | (구체적 동작으로 대체) |

## 사용자 검증 체크리스트

기능 설계 전 반드시 아래 4개를 문서화:

1. 사용자별 목표 완료율 목표(예: 상담사 80%가 1회 입력으로 문의 분류 완료)
2. 실패 허용조건(예: 상담 저장 실패 시 1회 재시도 가능)
3. 혼란 포인트(예: 유형/채널 중복 선택)
4. 대체 동선(예: 네트워크 불안정 시 임시 저장 후 재전송)

## 페르소나 작성 예시 (AICC 도메인 — 두 축 적용 사례)

아래는 위 두 축(이용자·운영자)을 AICC 도메인에 적용한 **예시**다(상담사=현장 운영자, 상담관리자=관리자). **도메인이 다르면** 같은 방식으로 그 도메인의 이용자·운영자 페르소나를 새로 작성한다 — 이 페르소나를 그대로 쓰지 말 것.

### 페르소나 A: 상담사 정해린 (핵심 사용자)

```
나이/직업: 29세, 옴니채널 상담사 (콜·채팅·메신저 동시 응대)
디지털 숙련도: 중상 (업무 시스템 능숙, 단축키 활용)
핵심 목표: 인입된 문의를 빠르게 파악하고 정확히 분류·응대하고 싶다
가장 큰 불안: "이전 상담 내역을 못 찾아 같은 걸 또 물어보게 될까 봐"
사용 맥락: 동시에 여러 건을 응대하며 듀얼 모니터로 상담 화면 조작
결정적 행동: 마우스보다 키보드 단축키로 빠르게 처리
```

→ **설계 원칙**: 핵심 액션 단축키 제공, 한 화면에서 이력·분류 동시 확인, 후처리 입력 최소화

### 페르소나 B: 상담관리자 한승우 (QA·운영 사용자)

```
나이/직업: 41세, 상담팀 관리자 (상담사 12명 관리)
디지털 숙련도: 상
핵심 목표: 상담 품질·처리량을 실시간으로 보고, 문제 상담을 빠르게 찾고 싶다
가장 큰 불안: "분류가 잘못돼도 모르고 지나가 지표가 왜곡될까 봐"
사용 맥락: 대시보드를 띄워 두고, 이상 지표 발생 시 개별 상담을 확인
결정적 행동: 지표 → 상담 상세로 바로 드릴다운
```

→ **설계 원칙**: 실시간 지표 가시화, 이상치 빠른 필터, 분류 오류 교정·재학습 동선 제공

## 페르소나 검증 질문

기능을 설계할 때마다 **두 축 모두** 통과하는지 물어라:

1. **이용자**가 자기 목표를 최소 단계로 달성할 수 있는가?
2. **운영자·관리자**가 이용자가 만든 것(예약·문의·주문 등)을 받아 처리·모니터링할 수 있는가?

한 축이라도 No면 설계를 수정한다.
(예시 — AICC: ① 상담사가 동시 응대 중 문의를 3초 내 파악 ② 관리자가 대시보드에서 문제 상담을 1단계로 드릴다운)

## 결정 근거 기록

각 사용자 판단 변경 시 `ADR` 형식으로 기록한다.

- 결정: 어떤 행동을 채택했는지
- 이유: 사용자 실패 비용 또는 구현 비용
- 대안: 버려진 대안과 거절 이유
