# Skill 03: 구조화 능력

## 목적
복잡한 요구사항을 계층적으로 정리하고,
기능 간 의존관계를 파악해 올바른 순서로 배치하는 능력.

---

## 실행 프로토콜

### Step 1: 요구사항 ID 생성

요구사항을 수집한 뒤 즉시 ID를 부여한다.

- 형식: `REQ-001`, `REQ-002`
- ID는 `PRD`, `IA`, `Flow`, `Screen`, `Test`에서 동일하게 사용
- 이름 변경이 생겨도 ID는 유지하고 `요구사항 레지스트리`에서 매핑만 수정

### Step 2: 기능 트리 생성 규칙

요구사항을 받으면 반드시 이 **계층 구조**로 분류해라 (괄호 안 값은 AICC 예시 — 도메인에 맞게 교체):

```
서비스
└── 도메인 (예: 상담 인입 / 상담 처리 / 이력 / 분석 / 설정)
    └── 기능 그룹 (예: 문의 분류 / 상담 응대 / 이력 조회)
        └── 세부 기능 (예: 유형 선택 / 채널 필터 / 상담사 배정)
            └── 엣지케이스 (예: 분류 불가 문의 / 배정 가능한 상담사 없을 때)
                └── 요구사항 ID 연결 (REQ-###)
```

### Step 3: 기능 의존관계 및 우선순위

기능 목록이 나오면 반드시 순서를 정해라.

- 질문: "A 기능이 없으면 B 기능이 동작하는가?"
  - 의존하면 → A가 먼저
  - 독립적이면 → 병렬 진행 가능
- 기능별 우선순위를 붙인다: `MUST`, `SHOULD`, `NICE`
- MUST 비율이 40%를 넘으면 과설계 조정

### Step 4: MVP 경계선 + 책임 주체

MVP 포함 기준 (없으면 서비스 불가)
- 핵심 가치 전달 플로우가 막히는 기능
- "고객이 목표 행위를 완료할 수 없다"면 MUST
- 법적/보안 필수 기능

MVP 이후 기준 (있으면 좋음)
- 편의 기능, 개인화, 통계 대시보드
- 발생 빈도 5% 미만 엣지케이스 처리
- 2차 사용자(간접 관계자) 전용 기능 — 단, **운영자·어드민은 여기서 제외한다**(스킬 02 기준 운영자는 이용자와 함께 1차 축이므로 MVP에 포함)

### Step 5: 구조화 산출물과 테스트 연계

각 기능 항목에 아래를 필수로 넣는다:
- 요구사항 ID
- 의존 관계
- 엣지케이스 1개 이상
- 검증 시나리오(성공/실패)
- 오너/담당(기획/디자인/개발/QA)

**구조화 산출물 포맷**

```markdown
## 기능 구조

### 도메인: [이름]

**MVP (v1)**
- [기능명] (REQ-###, 우선순위: MUST/SHOULD/NICE)
  - 의존: [선행 기능]
  - 엣지케이스: [주요 1개]
  - 검증: [성공 조건 / 실패 조건]

**v2 예정**
- [기능명] (REQ-###, 우선순위: SHOULD/NICE)

**보류/미포함**
- [기능명]: [보류 이유]
  - ADR: [근거]
```

## 기능 도메인 맵 예시 (AICC 도메인)

아래는 AICC 도메인의 기능 지도 **예시**다. 기능 요청이 오면 도메인 맵에서 위치를 먼저 확인하되, **도메인이 다르면 그 도메인의 맵으로 교체**한다:

```
[상담 인입 도메인]      [상담 처리 도메인]
- 채널 통합 인입        - 문의 유형 분류
- 문의 대기열           - 상담 응대/메모
- 상담사 배정/이관      - 상담 저장/종료
- 우선순위/SLA          - 후처리 코드 입력

[고객/이력 도메인]      [운영 도메인]
- 고객 식별/통합        - 유형 체계 관리
- 상담 이력 조회        - 채널 연동 설정
- 이전 문의 요약        - 배정 규칙 설정
- 동일 문의 병합        - 권한/조직 관리

[분석 도메인]          [품질(QA) 도메인]
- 채널별 인입량         - 분류 정확도 모니터링
- 유형별 처리량         - 상담 품질 평가
- 응대/대기 시간        - 오분류 교정/재학습
- 상담사별 실적
```
