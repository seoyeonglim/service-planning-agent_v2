# 서비스 기획 에이전트 (service-planning-agent_v2)

AI 코딩 에이전트(Claude Code / Codex CLI)를 **"10년 경력의 시니어 서비스 기획자"**로 동작시키는 에이전틱 워크플로우입니다.

핵심 아이디어는 하나입니다 — **AI가 문서를 바로 쓰지 못하게 막고, 실제 기획자처럼 "질문 → 확인 → 산출물 → 검증" 순서를 강제하는 것.** 이를 위해 세 겹의 장치로 구성됩니다:

1. **`CLAUDE.md`** — 전체 워크플로우 헌법 (Phase 0~4 순서, 게이트, 금지사항)
2. **`.claude/skills/` 13개 스킬** — 각 단계의 상세 실행 매뉴얼 (CLAUDE.md가 단계마다 "이 파일을 읽어라"고 지시)
3. **훅(hook) + 파이썬 스크립트** — 규칙 준수를 사람이 아닌 기계가 감시 (추적성 자동 검증, 스킬 동기화)

모든 산출물은 추적 ID로 사슬처럼 연결되며, 이 사슬이 끊기면 다음 단계로 진행할 수 없습니다.

```
REQ-###  요구사항   →  EC-##   엣지케이스
   ↓                    ↓
SC-##    화면      →  TC-###  테스트 케이스
   ↓
FS-###   기능명세   →  WBS 주차 배치
```

## 폴더 구조

폴더는 크게 **세 덩어리**로 나뉩니다:

| 덩어리 | 위치 | 한 줄 설명 |
|---|---|---|
| 📖 규칙서 | `CLAUDE.md`, `AGENTS.md`, `.claude/skills/` | AI에게 "이 순서로, 이렇게 일해라"를 알려주는 매뉴얼 |
| ⚙️ 자동 도구 | `.claude/scripts/`, `.claude/hooks/`, 설정 파일 | 규칙이 지켜졌는지 자동으로 검사·정리해주는 프로그램 |
| 📂 작업 결과물 | `docs/` | 실제 프로젝트에서 만들어진 기획 문서·화면들 |

### 📖 규칙서 — AI가 읽는 매뉴얼

```
CLAUDE.md                      # 전체 작업 순서와 규칙 (가장 중요한 파일)
AGENTS.md                      # 위와 같은 내용의 Codex(다른 AI 도구)용 복사본
.claude/skills/
├── 01~13_*.md                 # 단계별 상세 매뉴얼 13개
│                              #   예) 01번 = 문제 정의하는 법, 13번 = 일정표 짜는 법
└── visual_generation/         # 와이어프레임·도식 그리는 법
```

> AI는 작업 단계마다 해당 번호의 매뉴얼을 펼쳐 읽고 그대로 따릅니다.

### ⚙️ 자동 도구 — 사람 대신 검사해주는 프로그램

```
.claude/scripts/
├── validate_traceability.py   # "빠뜨린 요구사항 없나?" 자동 검사기
├── sync_codex_skills.py       # 매뉴얼 수정하면 Codex용 복사본도 자동 갱신
├── make_pdf.sh                # 완성된 기획서를 고객 전달용 PDF로 변환
└── _mermaid_to_img.py         # (PDF 만들 때 다이어그램을 이미지로 바꿔주는 보조)

.claude/hooks/session_start.sh # 시작할 때 "지금 어디까지 했더라?" 현황판 출력
.claude/settings.local.json    # 위 검사기들이 자동으로 돌아가도록 연결하는 설정
.claude/assets/pdf/            # PDF의 글꼴·색상 통일 규칙
.codex/skills/                 # 자동 생성된 복사본 (직접 수정 금지!)
.mcp.json                      # 고객이 준 워드/엑셀/PPT 파일을 읽을 수 있게 하는 설정
```

### 📂 작업 결과물 — 프로젝트마다 하나씩 생기는 폴더

프로젝트를 시작하면 `docs/` 아래에 프로젝트 폴더가 생기고, 작업이 진행될수록 하위 폴더가 차례로 채워집니다. **폴더 이름이 곧 작업 단계**예요:

```
docs/[프로젝트명]/
├── prd/                # 1단계: 뭘 만들지 정의한 기획서 (PRD)
├── ui/                 # 2단계: 화면 설계 문서 4종 (구조도·플로우·화면명세·디자인)
│   └── screens/        # 3단계: 실제로 열어볼 수 있는 HTML 화면
├── assets/wireframes/  # 3단계: 화면 스케치(와이어프레임)와 설명 달린 도해
├── fnspec/             # 4단계: 개발팀에게 넘기는 기능명세서
├── wbs/                # 4단계: 누가 언제 뭘 하는지 일정표 (WBS)
├── sow/                # 계약용 작업명세서
└── ref/                # 고객사가 준 자료 (RFP, 회의록 등) — 참고용 원본
```

> 이 폴더는 고객사 정보가 담기므로 git(GitHub)에는 올라가지 않고 내 컴퓨터에만 보관됩니다.

## Phase별 워크플로우

```
Phase 0  프로젝트 경로 설정   → 프로젝트명 확인, docs/[프로젝트명]/ 생성
Phase 1  PRD 작성            → 스킬 01→02→03→04→06→05 순서로 실행
Phase 2  UI 문서 4종          → 스킬 07→08→09→10
Phase 3  UI 화면 생성          → 스킬 11 + visual_generation
Phase 4  개발 핸드오프         → 스킬 12(기능명세서) → 13(WBS)
```

- Phase 3과 Phase 4는 서로 의존하지 않아 **병행 가능**합니다.
- 각 Phase의 종료에는 **이중 게이트**가 있습니다:
  1. **사용자 승인** — "맞아요" 확인 전까지 다음 단계 진행 금지
  2. **자동 검증** — `python3 .claude/scripts/validate_traceability.py --strict` 실행, ❌ 오류 0이어야 통과

### 공통 운영 규칙 (모든 Phase)

- 각 기능은 `REQ-###` ID로 관리하고, 이름이 바뀌어도 ID는 유지
- 제안한 규칙/결정은 **ADR(결정사항 로그)** 형태로 기록 — 배경/대안/선택/이유/영향
- **버전 생애주기**: 초안(v0.x)은 git 커밋으로 변경 추적(문서 내 변경 이력 표 선택), v1.0 확정 이후엔 변경 1건마다 **변경 기록(CR-###)**을 `prd/CHANGELOG.md`에 작성 — 대상 / AS-IS(변경 전 문구) / TO-BE(변경 후 문구) / 변경 범위 / 사유(ADR 링크) / 승인 날짜 6항목이 한 레코드에 보이게. 문서 내 변경 이력 표에는 요약 행만 누적
- 미결(❓) 항목이 남아 있으면 다음 단계 진입 금지
- 선택지가 있는 질문은 반드시 `AskUserQuestion` 도구 사용 (선택지 2~4개, 복수 선택 시 multiSelect)

### 공통 금지사항

- 질문 없이 바로 문서 작성 금지
- PRD 없이 UI 문서, UI 문서 없이 UI 생성, 화면 명세 없이 기능명세서, 기능명세서 없이 WBS 작성 금지
- 기획 문서에 기술 구현 방법 기재 금지
- 추상적 표현 금지 — "편리한", "빠른", "쉬운" → 구체적 수치·조건으로 대체
- 영문 더미 데이터 사용 금지 (모든 UI 라벨·더미 데이터는 한국어)
- 외부 이미지 생성 API 호출 금지 — 모든 시각 산출물은 HTML/Mermaid로 직접 작성

## 스킬 상세

번호 스킬의 원본은 `.claude/skills/NN_name.md` **하나뿐**이며, `.codex/skills/`는 자동 생성물입니다.

### Phase 1: PRD 만들기 (01~06)

번호와 실행 순서가 다른 것이 특징입니다. **01 → 02 → 03 → 04 → 06 → 05** 순서로, 테스트 케이스(06)를 먼저 도출한 뒤 문서(05)를 조립합니다.

#### 01. 문제 정의력 (`01_problem_definition.md`)

사용자가 요청한 솔루션("~기능 만들어줘")이 아니라 그 뒤에 숨은 **진짜 문제**를 먼저 찾아내는 스킬.

- **Step 1 — 5 Whys**: 아이디어에 "왜?"를 최대 5번 질문. 4번째 Why쯤에서 진짜 문제가 드러난다는 원칙
- **Step 2 — 문제 정의 수집 질문 5개** (반드시 이 순서): ① 지금 어떻게 해결하고 있나(현재 해결책) ② 얼마나 자주 발생하나(빈도/규모) ③ 구체적 피해는(임팩트 수치화) ④ 해결되면 어떤 상태를 원하나(성공 기준) ⑤ 증거는(로그/VOC/통계 등 근거)
- **Step 3 — 문제 정의서 표**: AS-IS / 발생 빈도 / 구체적 피해 / TO-BE / 측정 가능한 성공 기준 / 증거 소스 / REQ ID — 7항목. 사용자가 "맞아요" 하기 전까지 진행 금지
- **Step 4 — PRD 진입 전 승인 체크 4항목**: 문제 진술 한 문장 요약 가능 / 피해가 최소 1개 수치로 표현 / TO-BE가 검증 가능한 사용자 행동 / 근거 소스 1개 이상
- 말미에 AICC(AI 컨택센터) 도메인 특화 "표면적 요청 → 진짜 문제" 매핑 표 포함

#### 02. 사용자 관점 전환 (`02_user_perspective.md`)

기획자 본인이 아니라 **실제 사용자 맥락**에서 모든 결정을 검증하는 스킬.

- **사용자 파악 질문 7개** (기획 시작 전 필수): 연령대, 사용 시점·장소, 시스템 숙련도, 사용 직전/직후 행동, 가장 두려운 실패 시나리오, 현재 대체 중인 과정 등
- **추상어 금지 번역표** (핵심 규칙): "간편하게"→"3번 탭 이내로", "빠르게"→"2초 이내에", "쉽게 찾을 수 있는"→"메인 화면에서 1단계 접근". "편리한 UI"/"직관적인" 등은 아예 금지
- **페르소나 2명을 판단 기준으로 강제**:
  - 상담사 정해린(29세, 옴니채널, 핵심 사용자) — 동시 응대·단축키 위주. 불안: "이전 상담 내역을 못 찾을까 봐"
  - 상담관리자 한승우(41세, 12명 관리, QA·운영) — 불안: "오분류로 지표 왜곡"
- **페르소나 검증 질문**: 기능마다 "정해린이 동시 응대 중 3초 안에 파악 가능한가?" + "한승우가 1단계로 드릴다운 가능한가?" — **둘 다 Yes가 아니면 설계 수정**
- 결정 근거는 ADR 형식(결정/이유/버려진 대안과 거절 이유)으로 기록

#### 03. 구조화 능력 (`03_structuring.md`)

요구사항을 계층으로 정리하고 의존관계를 파악해 MVP 경계를 확정하는 스킬.

- **REQ ID 체계**: `REQ-001` 형식, PRD/IA/플로우/화면/테스트 전체에서 동일 ID 사용
- **기능 트리 5계층**: 서비스 → 도메인 → 기능 그룹 → 세부 기능 → 엣지케이스, 각각 REQ 연결
- **우선순위 3단계**: `MUST / SHOULD / NICE` — **MUST 비율이 40%를 넘으면 과설계로 판단하고 조정**
- **MVP 경계 기준**: 포함 = 핵심 가치 플로우를 막는 기능·법적/보안 필수 / 제외 = 편의 기능·개인화·통계 대시보드·발생 빈도 5% 미만 엣지케이스
- 산출물은 도메인별 "MVP(v1) / v2 예정 / 보류·미포함(+ADR 근거)" 3구획으로 정리. 각 기능에 의존 관계·엣지케이스 1개 이상·검증 시나리오·오너 필수
- AICC 기능 도메인 맵 6개(상담 인입/처리/고객·이력/운영/분석/품질 QA) 포함 — 기능 요청이 오면 지도에서 위치부터 확인

#### 04. 엣지케이스 감지 (`04_edge_cases.md`)

정상 플로우의 구멍을 개발 전에 찾아내는 스킬 (개발 후 발견 시 비용 10배, 서비스 중 발견 시 100배라는 전제).

- **우선순위 분류 축**: 영향도 × 빈도 × 대응 난이도 → MUST/SHOULD/NICE
- **범용 체크리스트 6개 카테고리 22개 질문** (모든 기능에 적용 필수):
  - 동시성(3) — 동일 자원 동시 점유, 저장 2회 연타, 설정 수정과 사용의 동시 발생
  - 네트워크/시스템(3) — 저장 직전 종료, 전송 중 서버 다운, 연동 단절
  - 데이터 경계값(3) — 특수문자·이모지, 잘못된 형식, 글자 수 초과
  - 시간/날짜(4) — 자정 넘는 집계, 영업시간 외, SLA 직전, 종료 건 재오픈
  - 상태 변화(4) — 담당자 이탈, 사용 중 코드 삭제, 채널 전환, 규칙 변경
  - 권한/접근(3) — 세션 만료 중 저장, 타 팀 조회, 탈퇴 데이터 처리
- **EC 문서 포맷**: `EC-[번호]: [상황명] (REQ-###)` 헤딩 + 7행 표(발생 조건/빈도/영향/처리 방법[무시·안내·자동 복구·강제 중단 중 택1]/우선순위/처리 화면/MVP 포함 여부)
- **게이트**: MUST 엣지케이스 하나라도 누락 시 UI 문서 단계 진입 금지

#### 05. 문서 작성 노하우 (`05_document_writing.md`)

읽는 사람이 질문 없이 이해하고 바로 행동할 수 있는 문서를 쓰는 스킬.

- **기본 원칙 5개**: 결론 우선(피라미드), 한 문장 한 의미, 추상어 삭제→수치 대체, ADR 기록, ID 중심 구조
- **독자별 전략**: 개발자(조건·예외·데이터 형식, 상세) / 대표·클라이언트(임팩트·수치·일정, 1페이지) / 디자이너(감정·플로우·상태 중심)
- **PRD 필수 템플릿 15개 섹션** (이 파일의 핵심): 문서 정보(버전·상태) → 한줄 요약 → 문제 정의(AS-IS/TO-BE/성공 기준/증거) → **요구사항 레지스트리 표** → 스코프 → 사용자 스토리 → 기능 상세 → NFR → 결정 로그(ADR) → 가정✅/확정🔲/미결❓ → 테스트 케이스 초안 → 변경 이력 → 관련 문서 추적
- **Mermaid 플로우차트 규칙**: happy path 먼저, 분기점마다 조건 명시, 오류 플로우 별도, 단계마다 액터 명시
- **발행 전 자가 검토 7문항** — 하나라도 No면 즉시 수정 (개발자가 질문 없이 구현 가능한가, 대표가 30초에 파악 가능한가 등)
- **변경 기록(CR) 규칙**: v1.0 확정 이후의 모든 변경은 `prd/CHANGELOG.md`에 CR-### 레코드로 기록 — AS-IS/TO-BE 문구 그대로 인용, 변경 범위는 추적성 검증으로 누락 확인, 방향성 결정은 ADR과 상호 링크(ADR=의사결정 기록, CR=실행 기록)

#### 06. 테스트 케이스 도출 (`06_test_cases.md`)

확정된 REQ·EC를 검증 가능한 형태(Given-When-Then)로 **번역**하는 스킬. 04 완료 후, 05로 PRD 조립하기 **직전**에 수행합니다.

- **핵심 원칙**: ① TC는 창작이 아니라 번역 ② Then은 반드시 관찰 가능·수치화("잘 동작한다" ❌ → "3초 이내 응답 시작" ✅) ③ **TC 헤딩 괄호에 검증 대상 REQ/EC 명시** — `### TC-001 (REQ-001, EC-03)` — 추적성 스크립트가 이 괄호를 읽어 커버리지를 자동 계산 ④ 우선순위는 검증 대상 REQ에서 상속
- **커버리지 목표**: 모든 MUST REQ → 정상 TC 최소 1개 / MUST·SHOULD EC → 예외 TC 최소 1개 / 측정값 있는 NFR → 성능 TC 최소 1개
- 예외 TC는 EC의 "발생 조건"=Given, "처리 방법"=Then으로 그대로 옮김
- `validate_traceability.py --strict`에서 "TC 없는 MUST REQ" ❌가 0이 될 때까지 반복

### Phase 3: UI 생성 (11 + visual_generation)

#### 11. UI 생성 (`11_ui_generation.md`)

화면 명세서 + 디자인 방향을 바탕으로 실제 동작하는 샘플 HTML을 생성하는 스킬.

- **시작 전 필수**: `ui/` 4개 문서 모두 읽기 — 하나라도 없으면 생성 시작 금지
- **화면당 5단계 절차** (본 HTML 바로 생성 금지, 각 단계마다 사용자 확인):

  ```
  ① Mermaid 구조도            → 확인
  ② 저해상도 HTML 와이어프레임  → 확인   (그레이스케일, Type B)
  ③ 주석 HTML 도해(콜아웃)     → 확인   (동작 규칙·조건·분기 설명, Type C)

  ⑤ 다음 화면으로
  ```

- **구현 강제 항목**: 동작하는 링크 이동, 실제 유효성 메시지 텍스트, 빈/로딩/에러 상태 최소 1개씩, 실패 시 복구 버튼(재시도/뒤로가기)
- **더미 데이터 고정 명단**: 고객 김지수·박서연·이민수·최유리·정하준 / 상담사 정해린·박도윤·김서아 / 관리자 한승우 — **영문 사용 금지**
- 각 화면 파일 상단에 REQ/SC/TC 추적 주석 명시, 화면 이동은 파일 간 링크(`location.href`)로 실제 클릭 가능하게
- 검수: TC 기준 핵심 시나리오 1개 이상을 화면에서 수동 실행 가능해야 함

#### visual_generation (`SKILL.md` + `REFERENCE.md`)

시각 자료를 외부 이미지 생성 API 없이 HTML/Mermaid로 직접 만드는 스킬. 13개 번호 스킬과 달리 프론트매터가 수작업 커스텀(`disable-model-invocation: true` — 명시 호출 전용)이라 동기화 대상에서 제외됩니다.

- **3가지 생성 유형**:
  | 유형 | 산출물 | 도구 | 용도 |
  |---|---|---|---|
  | Type A | 서비스 플로우 다이어그램 | Mermaid (문서 인라인) | PRD/플로우 보완, 노드 최대 10개 |
  | Type B | UI 컨셉 와이어프레임 | 그레이스케일 HTML | 레이아웃 확인 — 일부러 "미완성 컨셉" 룩 유지 |
  | Type C | 주석 HTML 도해 | HTML + 절대위치 콜아웃 | 동작 규칙·조건·분기를 번호 뱃지(①②③)로 설명 |
- **콜아웃 색상 규약 5색**: 빨강=오류·제약·비활성 / 파랑=사용자 액션·이동 / 초록=활성·완료·정상 / 주황=경고·트리거 / 회색=기본 동작·닫힘
- 저장 규칙: `docs/[프로젝트명]/assets/wireframes/wf-[SC-ID]-[상태명].html`, `annotated-[SC-ID]-[라벨].html` — 해당 문서에 상대경로 링크 연결
- 콜아웃·라벨 반드시 한국어, Type C 콜아웃 텍스트는 기획서 동작 규칙 문구와 일치시킴

### Phase 4: 개발 핸드오프 (12~13)

#### 12. 기능명세서 (`12_functional_spec.md`)

PRD(서술)와 화면 명세(레이아웃) 사이의 빈칸인 **동작 규칙**을 `FS-###` 단위로 정의하는 개발·QA 핸드오프 문서. Phase 3(UI 생성)과 병행 가능.

- **핵심 원칙 6개**:
  1. FS는 창작이 아니라 **분해** — REQ 1개→FS 여러 개, PRD에 없는 기능을 FS에서 새로 만들지 않음(스코프 크립 방지)
  2. 모든 FS는 **입력→처리 규칙→출력→예외** 4칸 완결 — 하나라도 못 채우면 미결(❓)
  3. 모든 FS에 REQ/SC/TC 연결 명시
  4. 조건 3개 이상 얽힌 분기는 별도 **결정표(D-#)**로 분리
  5. 전 기능 공통 규칙(인증·마스킹·폴백 등)은 공통 규칙 섹션으로 승격 (같은 문장 3번째 반복 시)
  6. **기술 구현 방법 기재 금지** — "무엇이 일어나야 하는가"까지만
- 실행 7단계: 범위·독자 확인 질문 → REQ 체크리스트화(MUST 커버리지 100%) → REQ→FS 분해 → 공통 규칙 분리 → 결정표 분리 → 추적 연결(우선순위는 REQ에서 상속, 임의 조정 금지) → 역추적표 + `--strict` 검증
- 산출물: `fnspec/기능명세서.md` — 기능 명세 표(`FS-ID|기능명|처리 규칙|입력|출력|예외(EC)|우선순위|연결`) + 결정표 + 역추적표

#### 13. WBS 작성 (`13_wbs.md`)

확정된 스코프(REQ·FS)를 **일정과 리소스**로 변환하는 스킬. 기능명세서 없이 작성 금지.

- **핵심 원칙 5개**:
  1. **명목일이 아닌 가용일로 계획** — 실제 달력(주말·법정공휴일·대체공휴일) 대조 필수
  2. **작업 단위 발명 금지** — Epic=REQ, Story=FS. FS에 없는 작업(인프라 등)은 별도 섹션+근거
  3. **외부 의존(전제)은 지뢰** — 🔴 표기 + "미제공 시 어느 주차로 후퇴하는가" 사전 정의
  4. **대외 약속 불변, 내부 목표 선행** — 고객사 게이트는 고정, 내부 목표를 1~2주 앞당겨 버퍼 확보
  5. 임의 산정값은 "(가정, 착수 시 확정)" 명시
- 두 개의 판: **기능판**(FS를 몇 주차에, 티켓 생성 기준 — 의존 FS가 선행 FS보다 앞 주차 배치 금지) + **리소스판**(멤버×주차 투입률 %, 합계 100% 초과 = 배치 오류)
- 게이트 통과 기준은 **수치화 필수** ("검수" ❌ → "Intent 정확도 75%·SOP 사인오프" ✅)
- 가용일 3~4일인 주는 ⚠️ 표시 + 용량 축소, 버퍼 주에는 기능 배치 금지

## 추적성 검증의 동작 원리

`validate_traceability.py`는 한 마디로 **"기획 문서들끼리 서로 약속한 ID가 다 맞는지 대조하는 채점기"**입니다. 사람이 눈으로 "REQ-007이 화면에도 있었나...?" 하고 찾는 일을 기계가 대신해줍니다. 잡아내는 것은 두 방향입니다:

- **고아** — 정의만 하고 어디서도 안 쓴 것 (기획서에만 있고 화면에서 사라진 기능 = 구현 누락 위험)
- **유령** — 쓰고 있는데 정의가 없는 것 (문서가 존재하지 않는 요구사항을 참조)

### 실행 방법

```bash
python3 .claude/scripts/validate_traceability.py                    # 그냥 검사 결과만 출력
python3 .claude/scripts/validate_traceability.py --strict           # 오류 있으면 "실패" 처리 (게이트용)
python3 .claude/scripts/validate_traceability.py --strict --report  # + 결과를 파일로도 저장
```

외부 라이브러리 없이 파이썬 기본 기능(정규식)만 사용하므로, 파이썬만 있으면 어디서든 돌아갑니다.

### ① 문서를 전부 읽어들인다

`docs/` 아래 프로젝트 폴더마다 `prd/*.md`(기획서), `ui/01~04*.md`(화면 설계 문서 4종), `ui/screens/*.html`(실제 화면)을 읽습니다. PRD가 없는 폴더는 "아직 시작 안 한 프로젝트"로 보고 건너뜁니다.

### ② 문서에서 ID를 두 종류로 긁어모은다

**(a) "정의된 ID"** — 정해진 자리에 적힌 것만 인정:

| ID | 어디에 적어야 정의로 인정되나 |
|---|---|
| `REQ-###` (요구사항) | PRD 레지스트리 표의 행: `\| REQ-001 \| 상담 자동분류 \| MUST \|` — 이때 MUST/SHOULD/NICE 우선순위도 같이 읽음 |
| `EC-##` (엣지케이스) | PRD의 제목 줄: `### EC-01: 동시 접속 충돌` |
| `TC-###` (테스트) | PRD의 제목 줄: `### TC-001 (REQ-001)` |
| `SC-##` (화면) | 화면 명세서의 제목: `## 채팅 화면 (SC-03)` + 인덱스 표 |

**(b) "언급된 ID"** — 모든 문서 본문을 정규식으로 훑어 ID가 나오는 곳을 전부 수집합니다. `REQ-001~005`(범위)나 `REQ-070·071·074`(나열) 같은 축약 표기도 개별 ID로 풀어서 셉니다.

그리고 **TC 제목의 괄호를 특별 취급**합니다. `### TC-001 (REQ-001, EC-03)`이라고 쓰면 "TC-001은 REQ-001과 EC-03을 검증하는 테스트"라는 연결 정보로 읽습니다. 스킬 06에 "괄호 안에는 검증 대상 ID만 적어라"는 규칙이 있는 이유가 이것입니다.

### ③ 두 목록을 대조한다 — 검사 4가지

비유하면 **출석부(정의)와 실제 출석(언급)을 맞춰보는 것**입니다:

1. **유령 찾기** — 본문에서 언급되는데 레지스트리에 없는 REQ → ❌
   *예: 화면 명세서가 REQ-099를 언급하는데 PRD에 REQ-099 정의가 없음*
2. **테스트 빠진 필수 기능 찾기** — MUST 요구사항인데 어느 TC 괄호에도 안 나옴 → ❌
   *"필수 기능은 반드시 테스트 1개 이상" 규칙 위반*
3. **고아 찾기** — PRD에 정의됐는데 어느 화면/플로우 문서에도 안 나오는 MUST REQ → ❌
4. **화면 HTML 검사** — HTML 파일 상단 주석에 적힌 REQ가 PRD에 실존하는지 → 없으면 ❌

포인트: 검사 범위는 **폴더에 뭐가 있느냐에 따라 자동으로 늘어납니다.** 아직 안 만든 단계를 미리 혼내지 않아요.

| 검사 | 실행 조건 | ❌ 오류 (게이트 차단) | ⚠️ 경고 (보완 대상) |
|---|---|---|---|
| ① PRD 내부 정합성 | PRD만 있어도 | 레지스트리에 없는 REQ 유령 참조 | 정의 없는 EC 참조 |
| ② TC 커버리지 | PRD만 있어도 | TC 없는 MUST REQ | TC 없는 EC, TC 자체가 없음 |
| ③ REQ↔화면 매핑 | `ui/` 생기면 | 화면에 없는 MUST REQ(고아), UI가 참조하나 PRD에 없는 REQ | 화면에 없는 SHOULD/NICE, 미정의 SC 참조 |
| ④ HTML 참조 유효성 | `ui/screens/` 생기면 | 화면 HTML이 참조하나 PRD에 없는 REQ | — |

심각도는 두 등급 — **❌ 오류**(반드시 고쳐야 통과)와 **⚠️ 경고**(알려주지만 통과는 시킴).

### ④ 성적표를 출력한다

```
🗂 [kyobo_lifeplanet_ai_salesbot]
■ 선언 현황
   REQ 65개 (MUST 19) · EC 16개 · TC 28개 · SC 15개
■ TC 커버리지
   MUST REQ TC 커버리지: 19/19 (100%)
  ✅ 모든 MUST REQ가 1개 이상의 TC로 검증됨
  ⚠️ 화면/플로우에 아직 없는 SHOULD/NICE REQ: REQ-123, REQ-170, REQ-171
  └─ 결과: ⚠️ 경고 있음 (오류 0 · 경고 2)
```

- `--report`를 붙이면 이 내용이 `prd/_traceability_report.md` 파일로도 저장됩니다 (git 추적 제외).
- `--strict`를 붙였고 ❌ 오류가 1개라도 있으면 **종료코드 2**(= "실패했다"는 신호)로 끝납니다. 이것이 Phase 게이트의 핵심 — 이 실패가 사라질 때까지 다음 단계로 못 넘어갑니다.

### 언제 실행되나 — 두 가지 경로

1. **자동 (advisory 모드)** — `.claude/settings.local.json`의 Stop 훅 덕분에, 이 폴더에서 Claude가 응답을 마칠 때마다 옵션 없이 실행됩니다. 차단은 안 하고 현황만 계속 보여주는 **계기판** 역할.
2. **수동 (게이트 모드)** — 각 Phase를 끝내고 사용자에게 "맞아요" 확인받기 직전에 `--strict`로 실행, 오류 0을 확인해야 통과.

정리하면: **문서 읽기 → ID 수집(정의 vs 언급) → 4가지 대조 → 성적표 + 통과/실패 신호**, 이 4단계가 전부입니다.

## 사용 방법

1. 이 폴더에서 Claude Code(또는 Codex CLI)를 실행
2. 서비스/기능 아이디어를 입력 — 예) `"미용실 예약 관리 앱을 만들고 싶어요"`
3. 에이전트가 프로젝트명을 물은 뒤(Phase 0), 문제 정의 질문부터 시작(Phase 1)
4. 각 단계 산출물을 확인하고 "맞아요"로 승인하면 다음 단계로 진행
5. 기존 프로젝트 재진입 시 `docs/` 하위 폴더명을 말하면 이어서 진행 (현황판: `bash .claude/hooks/session_start.sh`)
