# 서비스 기획 에이전트 (service-planning-agent_v2)

Claude Code / Codex CLI를 **"10년차 시니어 서비스 기획자"**처럼 일하게 만드는 워크플로우입니다.

핵심은 하나 — **AI가 문서부터 쓰지 못하게 막고, 진짜 기획자처럼 "질문 → 확인 → 산출물 → 검증" 순서를 지키게** 하는 것.

이를 위한 세 가지 장치:

1. **`CLAUDE.md`** — 전체 규칙서 (작업 순서·게이트·금지사항)
2. **`.claude/skills/`** — 단계마다 읽는 상세 매뉴얼 15개
3. **훅 + 스크립트** — 규칙을 지켰는지 기계가 자동 검사

모든 산출물은 추적 ID로 사슬처럼 이어지고, **사슬이 끊기면 다음 단계로 못 넘어갑니다.**

```
REQ(요구사항)  →  EC(엣지케이스)
   ↓                 ↓
SC(화면)      →  TC(테스트)
   ↓
FS(기능명세)   →  WBS(일정)
```

## 폴더 구조

| 무엇 | 위치 | 설명 |
|---|---|---|
| 📖 규칙서 | `CLAUDE.md`, `AGENTS.md`, `.claude/skills/` | AI에게 "이 순서로 일해라"를 알려주는 매뉴얼 |
| ⚙️ 자동 도구 | `.claude/scripts/`, `.claude/hooks/` | 규칙이 지켜졌는지 검사·정리하는 프로그램 |
| 📂 결과물 | `docs/` | 실제로 만들어진 기획 문서 (프로젝트마다 독립 저장소) |
| 📚 사람용 설명 | `README.md`, `guide/` | 워크플로우가 어떻게 도는지 설명 |

> `docs/`는 고객 정보가 담겨 이 저장소엔 올라가지 않습니다. 프로젝트마다 그 자리에서 **별도의 비공개 git 저장소**로 따로 관리해요.

## 작업 순서 (Phase)

```
Phase 0  프로젝트 설정        → 프로젝트명 확인, docs/[프로젝트명]/ 생성
Phase 1  PRD 작성            → 스킬 01~06
Phase 2  UI 문서 4종          → 스킬 07~10
Phase 3  UI 화면 생성          → 스킬 11
Phase 4  개발 핸드오프         → 스킬 12(기능명세서) → 13(WBS)
운영     변경 관리·정합성 검증  → 스킬 14 · 15
```

각 단계 끝에는 **두 개의 문**이 있어, 둘 다 통과해야 다음으로 갑니다:

1. **사용자 승인** — "맞아요" 확인 전엔 진행 안 함
2. **자동 검증** — `validate_traceability.py --strict` 실행, 오류 0이어야 통과

> Phase 3과 4는 서로 의존하지 않아 **동시에 진행**할 수 있습니다.

### PRD 이후, 진행 모드를 먼저 고릅니다

PRD가 끝나면 곧바로 UI를 파는 게 아니라, **이번 사이클 목표가 뭔지 사용자에게 물어봅니다.** 비싼 산출물(기능명세·고해상도 UI)은 방향이 확정된 뒤에 만들자는 원칙이에요.

- **① 방향 확정·리뷰 (Quick)** — 경량 화면 명세 → 저해상도 와이어프레임까지만 만들어 방향·플로우를 확정하고 리뷰·싱크. 기능명세·고해상도 UI는 다음 사이클로 미룸.
- **② 전체 산출 (Full)** — 화면 명세 → 와이어프레임으로 방향 확정 → **기능명세(처리·예외 확정)** → 고해상도 UI → WBS. 개발 핸드오프까지 다 뽑을 때.

> 어느 모드든 **화면 명세가 와이어프레임보다 먼저**입니다. Full 모드에선 **기능명세를 고해상도 UI 앞에** 두어, 뒤늦게 드러난 예외·상태가 UI에 반영되게 합니다.

**꼭 지키는 규칙 (전체는 CLAUDE.md 참조):**

- 기능은 `REQ-###` ID로 관리 — 이름이 바뀌어도 ID는 유지
- 질문 없이 문서 작성 금지, 순서 건너뛰기 금지
- 추상어("편리한", "빠른") 금지 → 구체적인 수치·조건으로
- UI 라벨·더미 데이터는 한국어만, 시각 자료는 HTML/Mermaid로 직접 작성

## 스킬 15개

단계별 실행 매뉴얼입니다. 상세 규칙은 **[guide/skills.md](guide/skills.md)** 참조.

| 단계 | 스킬 | 하는 일 |
|---|---|---|
| Phase 1 | 01 문제 정의 | 요청 뒤에 숨은 진짜 문제를 질문으로 찾기 |
| | 02 사용자 관점 | 페르소나를 판단 기준으로 삼기 |
| | 03 구조화 | REQ 채번, 기능 트리, 우선순위, MVP 경계 |
| | 04 엣지케이스 | 정상 흐름의 구멍 찾기 (EC) |
| | 05 문서 작성 | PRD 템플릿 조립 + 변경 기록 형식 |
| | 06 테스트 케이스 | REQ·EC를 테스트(TC)로 번역 |
| Phase 2 | 07 서비스 구조도 (IA) | REQ를 화면(SC)으로 변환 |
| | 08 사용자 플로우 | 정상·예외 경로를 Mermaid로 |
| | 09 화면 명세서 | 화면마다 데이터·동작·상태 명세 |
| | 10 디자인 방향 | 색·글꼴·레이아웃 값 확정 |
| Phase 3 | 11 UI 생성 | 화면당 4단계(구조도→와이어프레임→도해→HTML) |
| | visual_generation | 외부 API 없이 시각 자료 제작 |
| Phase 4 | 12 기능명세서 | REQ를 기능(FS)으로 분해 |
| | 13 WBS | 기능을 일정 주차에 배치 |
| 운영 | 14 변경 요청 처리 | 확정 후 변경·추가·삭제 반영 |
| | 15 정합성 검증 | 문서 간 충돌을 3단계로 적발 (게이트 통과 직전·대규모 변경 후 권장 · 개발 핸드오프 전 필수) |

## 자동 도구

**저절로 도는 것 (훅):**

- 응답이 끝날 때마다 → 추적성 현황을 보여줌 (차단은 안 함)
- 스킬을 고치면 → Codex용 사본을 자동 동기화
- 프로젝트 저장소에서 커밋할 때 → **커밋 게이트**가 추적성·정합성을 검사, 오류가 있으면 커밋을 막음
  - 개발팀에 넘기는 `[handoff]` 커밋은 **스킬 15(문장 의미 대조)를 수행했는지**까지 확인

**직접 실행하는 스크립트:**

| 스크립트 | 하는 일 |
|---|---|
| `validate_traceability.py` | ID **연결이 끊긴 곳** 적발 → [상세](guide/traceability.md) |
| `consistency_check.py` | 연결됐는데 **값이 어긋난 곳** 적발 (우선순위·배치 등) → [상세](guide/consistency.md) |
| `commit_gate.sh` / `install_commit_gate.sh` | 위 두 검사를 커밋 때 자동 실행하는 게이트 (본체·설치기) |
| `impact_scan.py` | "이 ID 바꾸면 어디를 봐야 하지?" 체크리스트 → [상세](guide/impact-scan.md) |
| `sync_codex_skills.py` | 스킬 원본 → Codex용 사본 생성 |
| `make_pdf.sh` | 기획서 → 고객 전달용 PDF |

> 커밋 게이트는 새 프로젝트를 `git init` 할 때 자동으로 설치됩니다. 기획 프로젝트가 아닌 폴더에선 아무 방해 없이 조용히 통과해요.

## Claude Code + Codex 양쪽 지원

- 지침: `CLAUDE.md`(원본) ↔ `AGENTS.md`(Codex용 사본) — 내용이 다르면 CLAUDE.md가 우선
- 스킬: `.claude/skills/` 원본만 수정 — `.codex/`는 자동 생성이라 직접 손대지 않음

## 사용 방법

1. 이 폴더에서 Claude Code(또는 Codex CLI) 실행
2. 아이디어 입력 — 예) `"미용실 예약 관리 앱을 만들고 싶어요"`
3. 프로젝트명을 물어본 뒤(Phase 0), 질문부터 시작(Phase 1)
4. 산출물을 확인하고 "맞아요"로 승인하면 다음 단계로
5. 기존 프로젝트 이어서 하려면 폴더명을 말하면 됩니다 (현황판: `bash .claude/hooks/session_start.sh`)

### 저장 위치

| 산출물 | 경로 |
|---|---|
| PRD | `docs/[프로젝트명]/prd/` |
| UI 문서 4종 | `docs/[프로젝트명]/ui/` |
| 화면 HTML | `docs/[프로젝트명]/ui/screens/` |
| 와이어프레임·주석 도해 | `docs/[프로젝트명]/assets/wireframes/` |
| 기능명세서 | `docs/[프로젝트명]/fnspec/` |
| WBS | `docs/[프로젝트명]/wbs/` |
| 회의록 | `docs/[프로젝트명]/meetings/` |
| 고객사 원자료 (로컬 전용) | `docs/[프로젝트명]/ref/` |

> **형상관리:** 프로젝트는 각각 독립 저장소로 관리해요. `prd·ui·fnspec·meetings·assets/wireframes`만 git으로 추적하고, 나머지(원자료·계약·내부 일정 등)는 로컬에만 둡니다. 커밋하면 게이트가 자동으로 검사하고, 푸시 전엔 고객 개인정보가 섞이지 않았는지 한 번 더 점검합니다.
