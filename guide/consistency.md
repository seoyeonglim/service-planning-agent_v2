# 내용 정합성 검증의 동작 원리 (consistency_check.py)

> 이 문서는 [README](../README.md)의 상세 가이드입니다. 짝이 되는 [추적성 검증 가이드](traceability.md)와 함께 읽으면 좋습니다.

`consistency_check.py`는 한 마디로 **"ID는 멀쩡한데 문서끼리 서로 다른 말을 하는 곳을 잡는 검사기"**입니다.

[추적성 검증](traceability.md)이 **"선이 이어졌나"**(ID 연결)를 본다면, 이 검사기는 **"이어진 선의 값이 맞나"**(연결된 문서의 속성값)를 봅니다. 예를 들어 어떤 기능이 근거 요구사항과 제대로 연결돼 있어도, 그 요구사항은 "반드시(MUST)"인데 기능명세엔 "있으면 좋음(SHOULD)"으로 적혀 있으면 — 추적성 검사는 통과시키지만 이 검사기가 잡아냅니다.

> ⚠️ **경계:** 이 검사기는 **기계가 확실히 판정할 수 있는 값 불일치**만 봅니다. 문장의 의미가 어긋나는 것(예: PRD "응답 3초 이내" vs 기능명세 "5초 이내")은 기계로 판정할 수 없어, 스킬 15(에이전트 의미 대조)가 담당합니다. 즉 정합성은 3계층입니다 — **추적성(연결) → consistency_check(값) → 스킬 15(문장 뜻)**.

## 실행 방법

```bash
python3 .claude/scripts/consistency_check.py                    # 검사 결과만 출력
python3 .claude/scripts/consistency_check.py --strict           # 오류 있으면 "실패" 처리 (게이트용)
python3 .claude/scripts/consistency_check.py --strict --report  # + 결과를 파일로도 저장
```

추적성 검사기와 마찬가지로 파이썬 기본 기능(정규식)만 사용합니다. ID를 긁어모으는 로직은 `validate_traceability.py`의 함수를 그대로 재사용합니다.

## ① 무엇을 읽나

프로젝트 폴더마다 아래를 읽어들입니다. PRD가 없는 폴더는 건너뜁니다.

| 문서 | 위치 | 무엇을 뽑나 |
|---|---|---|
| PRD 레지스트리 | `prd/*.md` | REQ별 **우선순위·추진 단계**(상속의 기준) |
| 기능명세서 | `fnspec/*.md` | FS별 **우선순위·단계·근거 REQ** |
| WBS | `wbs/*.md` | MUST FS가 배치됐는지 확인용 |
| 화면 명세서 | `ui/03_screen_spec.md` | 선언된 화면(SC) 목록 |
| 와이어프레임·본 HTML | `assets/wireframes/*.html`, `ui/screens/*.html` | 화면별 산출물 존재 여부 |

> `prd/`의 `_*.md`(자동 생성 리포트)와 `CHANGELOG.md`는 판정에서 제외합니다.

## ② 무엇을 판정하나 — 기계 판정 8가지

비유하면 **"연결된 전선 양 끝의 규격표를 대조하는 것"**입니다. 붙어는 있는데 값이 어긋난 곳을 찾습니다.

| 검사 | 무엇을 보나 | ❌ 오류 (게이트 차단) | ⚠️ 경고 (보완 대상) |
|---|---|---|---|
| ① FS ID 중복 | 같은 `FS-###`가 두 번 선언 | 중복 선언된 FS | — |
| ② FS 우선순위 상속 | FS 우선순위가 근거 REQ와 같은가 | 근거 REQ와 우선순위가 다른 FS | — |
| ③ FS 추진 단계 일치 | FS 단계(P1~)가 근거 REQ와 같은가 | — | 단계가 다른 FS / 근거 REQ를 못 찾은 FS |
| ④ MUST FS의 WBS 배치 | 필수 FS가 WBS에 실렸나 | WBS에 없는 MUST FS | — |
| ⑤ 화면 산출물 커버리지 | 화면마다 와이어프레임·본 HTML이 있나 | — | 와이어프레임 없는 화면 / 본 HTML 없는 화면 |
| ⑥ 폐기 REQ의 활성 참조 | 폐기된 REQ를 살아있는 문서가 쓰나 | 폐기됐는데 UI·FS가 참조 중인 REQ | — |
| ⑦ 화면 상호 링크 | 본 HTML이 명세·주석도해와 SC-ID로 이어지나 | — | 명세 링크(03_screen_spec.md) 없는 화면 / 실재하지 않는 주석도해를 가리키는 화면 |
| ⑧ REQ 중복 선언 (PRD 내부) | 요구사항 레지스트리에 같은 REQ가 두 번 정의·우선순위 상충하나 | — | 레지스트리에 중복 정의된 REQ / 우선순위 상충 REQ |

각 검사를 조금 더 풀면:

- **① 중복** — 기능명세서 안에 `FS-012`가 두 번 나오면 정의가 흩어져 어느 게 진짜인지 모호해집니다.
- **② 우선순위 상속** — FS는 근거가 되는 REQ에서 우선순위를 **물려받아야** 합니다. 임의로 올리거나 내리면 ❌. (근거 REQ가 여럿이면 그중 하나와만 같아도 통과)
- **③ 단계 일치** — FS의 추진 단계가 근거 REQ의 단계와 어긋나면 알려줍니다(경고). 근거 REQ 자체를 레지스트리에서 못 찾으면 그것도 경고.
- **④ WBS 배치** — "필수(MUST) 기능인데 일정표에 없다"면 빠뜨린 것이므로 ❌.
- **⑤ 산출물 존재** — 화면 명세서에 선언된 `SC-03`이 있으면, `assets/wireframes/`와 `ui/screens/`에 그 화면 파일이 실제로 있는지 파일명으로 확인합니다(없으면 경고).
- **⑥ 폐기 좀비** — 우선순위 칸이 `폐기`로 바뀐 REQ를 UI 문서나 기능명세서가 여전히 참조하면 ❌. (레지스트리 행 자체는 이력 보존을 위해 남기되, 활성 문서의 참조는 끊어야 함)
- **⑦ 화면 상호 링크** — 각 본 HTML(`ui/screens/*.html`)이 자기 상세 명세(`03_screen_spec.md`)와 주석도해(`annotated-*.html`)를 SC-ID로 가리키는지 확인한다. 명세 링크가 없거나, 가리키는 주석도해 파일이 실재하지 않으면 경고(⚠️). "화면 하나만 열어도 나머지를 따라갈 수 있게" 하는 스킬 11 규칙이 실제로 지켜졌는지 보는 검사 — 기존 프로젝트를 갑자기 막지 않도록 오류(❌)가 아닌 경고로 둔다.

## ③ 성적표를 출력한다

```
🗂 [프로젝트명]
■ ID 중복 선언
  ✅ 중복 선언 없음
■ FS 우선순위 상속 (근거 REQ에서 상속, 임의 조정 금지)
  ✅ FS 65행 전수 — 우선순위 상속 일치
■ MUST FS의 WBS 배치
  ✅ MUST FS 22건 전부 WBS에 배치됨
■ 화면(SC) 산출물 커버리지
  ✅ 화면 15개 전부 와이어프레임·본 HTML 존재
  └─ 결과: ✅ 통과 (오류 0 · 경고 0)
```

- `--report`를 붙이면 `prd/_consistency_report.md`로도 저장됩니다 (git 추적 제외).
- `--strict`를 붙였고 ❌ 오류가 1개라도 있으면 **종료코드 2**로 끝납니다 — 게이트 차단 신호.

## 왜 기계가 여기까지만 하나

이 검사기는 **표·ID·파일 존재처럼 "딱 떨어지는 값"**만 판정합니다. 반대로 아래는 기계가 못 잡습니다:

- PRD "3초" vs 기능명세 "5초" 같은 **수치 충돌**
- "실시간 처리" vs "배치 처리" 같은 **방식 충돌**
- 조건·예외 문장이 서로 모순되는 경우

이런 **문장 의미 충돌**은 스킬 15가 4영역 병렬 에이전트로 읽어 대조하고, 발견을 `[결정]`·`[기계적]`·`[상류 CR]`로 분류해 처리합니다. 그래서 스크립트 마지막에도 "문장 의미 충돌은 스킬 15로 별도 수행"이라고 안내합니다.

## 언제 실행되나

1. **수동 (게이트 모드)** — 게이트 통과 직전·대규모 변경 후 `--strict`로 실행. **개발 핸드오프 전에는 필수**입니다.
2. **커밋 게이트 (자동 차단)** — 프로젝트 저장소에서 커밋할 때 `commit-msg` 훅(`commit_gate.sh`)이 `validate_traceability.py --strict`에 이어 이 스크립트를 `--strict`로 돌립니다. ❌ 오류가 있으면 **커밋 자체가 막힙니다.** `[handoff]` 커밋은 스킬 15 수행 확인 토큰(`Skill15-Reviewed: yes`)까지 요구합니다.

> **⑧ REQ 중복 선언**은 **PRD 내부 정합성**의 기계 판정분이다 — `요구사항 레지스트리` 섹션 안에서 같은 REQ가 우선순위를 달고 두 번 이상 정의됐는지(중복), 그 우선순위가 서로 다른지(상충)를 본다. 오탐을 막으려고 **레지스트리 섹션 범위 안**만 검사한다(스코프·추적표 등 다른 표의 REQ는 정상적으로 여러 번 나오므로 제외). 문장 뜻의 자기모순·중복·산재는 기계 불가 → 스킬 15의 **PRD 내부 자기 검토(영역 E)**가 담당.

정리하면: **문서 읽기 → REQ/FS 속성 대조 → 8가지 값 검사 → 성적표 + 통과/실패 신호**. 추적성(연결)이 놓치는 "값 어긋남"을 메우고, 자신이 못 잡는 "문장 뜻"은 스킬 15로 넘깁니다.
