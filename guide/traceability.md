# 추적성 검증의 동작 원리 (validate_traceability.py)

> 이 문서는 [README](../README.md)의 상세 가이드입니다.

`validate_traceability.py`는 한 마디로 **"기획 문서들끼리 서로 약속한 ID가 다 맞는지 대조하는 채점기"**입니다. 사람이 눈으로 "REQ-007이 화면에도 있었나...?" 하고 찾는 일을 기계가 대신해줍니다. 잡아내는 것은 두 방향입니다:

- **고아** — 정의만 하고 어디서도 안 쓴 것 (기획서에만 있고 화면에서 사라진 기능 = 구현 누락 위험)
- **유령** — 쓰고 있는데 정의가 없는 것 (문서가 존재하지 않는 요구사항을 참조)

> 이 검사는 **ID 연결만** 봅니다. ID는 멀쩡한데 문서끼리 **내용**이 어긋난 경우의 판정은 문서 서열 규칙(CLAUDE.md 공통 운영 규칙: PRD > UI 문서 > 기능명세서·화면 HTML > WBS, 상류 우선·하류 수정)을 따릅니다.

## 실행 방법

```bash
python3 .claude/scripts/validate_traceability.py                    # 그냥 검사 결과만 출력
python3 .claude/scripts/validate_traceability.py --strict           # 오류 있으면 "실패" 처리 (게이트용)
python3 .claude/scripts/validate_traceability.py --strict --report  # + 결과를 파일로도 저장
```

외부 라이브러리 없이 파이썬 기본 기능(정규식)만 사용하므로, 파이썬만 있으면 어디서든 돌아갑니다.

## ① 문서를 전부 읽어들인다

`docs/` 아래 프로젝트 폴더마다 `prd/*.md`(기획서), `ui/01~04*.md`(화면 설계 문서 4종), `ui/screens/*.html`(실제 화면)을 읽습니다. PRD가 없는 폴더는 "아직 시작 안 한 프로젝트"로 보고 건너뜁니다.

## ② 문서에서 ID를 두 종류로 긁어모은다

**(a) "정의된 ID"** — 정해진 자리에 적힌 것만 인정:

| ID | 어디에 적어야 정의로 인정되나 |
|---|---|
| `REQ-###` (요구사항) | PRD 레지스트리 표의 행: `\| REQ-001 \| 상담 자동분류 \| MUST \|` — 이때 MUST/SHOULD/NICE 우선순위도 같이 읽음 |
| `EC-##` (엣지케이스) | PRD의 제목 줄: `### EC-01: 동시 접속 충돌` |
| `TC-###` (테스트) | PRD의 제목 줄: `### TC-001 (REQ-001)` |
| `SC-##` (화면) | 화면 명세서의 제목: `## 채팅 화면 (SC-03)` + 인덱스 표 |

**(b) "언급된 ID"** — 모든 문서 본문을 정규식으로 훑어 ID가 나오는 곳을 전부 수집합니다. `REQ-001~005`(범위)나 `REQ-070·071·074`(나열) 같은 축약 표기도 개별 ID로 풀어서 셉니다.

그리고 **TC 제목의 괄호를 특별 취급**합니다. `### TC-001 (REQ-001, EC-03)`이라고 쓰면 "TC-001은 REQ-001과 EC-03을 검증하는 테스트"라는 연결 정보로 읽습니다. 스킬 06에 "괄호 안에는 검증 대상 ID만 적어라"는 규칙이 있는 이유가 이것입니다.

## ③ 두 목록을 대조한다 — 검사 4가지

비유하면 **출석부(정의)와 실제 출석(언급)을 맞춰보는 것**입니다:

1. **유령 찾기** — 본문에서 언급되는데 레지스트리에 없는 REQ → ❌
   *예: 화면 명세서가 REQ-099를 언급하는데 PRD에 REQ-099 정의가 없음*
2. **테스트 빠진 필수 기능 찾기** — MUST 요구사항인데 어느 TC 괄호에도 안 나옴 → ❌
   *"필수 기능은 반드시 테스트 1개 이상" 규칙 위반*
3. **고아 찾기** — PRD에 정의됐는데 어느 화면/플로우 문서에도 안 나오는 MUST REQ → ❌
4. **화면 HTML 검사** — HTML 파일 상단 주석에 적힌 REQ가 PRD에 실존하는지 → 없으면 ❌

포인트: 검사 범위는 **폴더에 뭐가 있느냐에 따라 자동으로 늘어납니다.** 아직 안 만든 단계를 미리 혼내지 않아요.

| 검사 | 실행 조건 | ❌ 오류 (게이트 차단) | ⚠️ 경고 (보완 대상) |
|---|---|---|---|
| ① PRD 내부 정합성 | PRD만 있어도 | 레지스트리에 없는 REQ 유령 참조 | 정의 없는 EC 참조 |
| ② TC 커버리지 | PRD만 있어도 | TC 없는 MUST REQ | TC 없는 EC, TC 자체가 없음 |
| ③ REQ↔화면 매핑 | `ui/` 생기면 | 화면에 없는 MUST REQ(고아), UI가 참조하나 PRD에 없는 REQ | 화면에 없는 SHOULD/NICE, 미정의 SC 참조 |
| ④ HTML 참조 유효성 | `ui/screens/` 생기면 | 화면 HTML이 참조하나 PRD에 없는 REQ | — |

심각도는 두 등급 — **❌ 오류**(반드시 고쳐야 통과)와 **⚠️ 경고**(알려주지만 통과는 시킴).

## ④ 성적표를 출력한다

```
🗂 [kyobo_lifeplanet_ai_salesbot]
■ 선언 현황
   REQ 65개 (MUST 19) · EC 16개 · TC 28개 · SC 15개
■ TC 커버리지
   MUST REQ TC 커버리지: 19/19 (100%)
  ✅ 모든 MUST REQ가 1개 이상의 TC로 검증됨
  ⚠️ 화면/플로우에 아직 없는 SHOULD/NICE REQ: REQ-123, REQ-170, REQ-171
  └─ 결과: ⚠️ 경고 있음 (오류 0 · 경고 2)
```

- `--report`를 붙이면 이 내용이 `prd/_traceability_report.md` 파일로도 저장됩니다 (git 추적 제외).
- `--strict`를 붙였고 ❌ 오류가 1개라도 있으면 **종료코드 2**(= "실패했다"는 신호)로 끝납니다. 이것이 Phase 게이트의 핵심 — 이 실패가 사라질 때까지 다음 단계로 못 넘어갑니다.

## 언제 실행되나 — 두 가지 경로

1. **자동 (advisory 모드)** — `.claude/settings.local.json`의 Stop 훅 덕분에, 이 폴더에서 Claude가 응답을 마칠 때마다 옵션 없이 실행됩니다. 차단은 안 하고 현황만 계속 보여주는 **계기판** 역할.
2. **수동 (게이트 모드)** — 각 Phase를 끝내고 사용자에게 "맞아요" 확인받기 직전에 `--strict`로 실행, 오류 0을 확인해야 통과.

정리하면: **문서 읽기 → ID 수집(정의 vs 언급) → 4가지 대조 → 성적표 + 통과/실패 신호**, 이 4단계가 전부입니다.
