# 스킬 상세 (01~15 + visual_generation)

> 이 문서는 [README](../README.md)의 상세 가이드입니다. 각 스킬의 원본은 `.claude/skills/NN_name.md` **하나뿐**이며, `.codex/skills/`는 자동 생성물입니다.

## Phase 1: PRD 만들기 (01~06)

번호와 실행 순서가 다른 것이 특징입니다. **01 → 02 → 03 → 04 → 06 → 05** 순서로, 테스트 케이스(06)를 먼저 도출한 뒤 문서(05)를 조립합니다.

### 01. 문제 정의력 (`01_problem_definition.md`)

사용자가 요청한 솔루션("~기능 만들어줘")이 아니라 그 뒤에 숨은 **진짜 문제**를 먼저 찾아내는 스킬.

- **Step 1 — 5 Whys**: 아이디어에 "왜?"를 최대 5번 질문. 4번째 Why쯤에서 진짜 문제가 드러난다는 원칙
- **Step 2 — 문제 정의 수집 질문 5개** (반드시 이 순서): ① 지금 어떻게 해결하고 있나(현재 해결책) ② 얼마나 자주 발생하나(빈도/규모) ③ 구체적 피해는(임팩트 수치화) ④ 해결되면 어떤 상태를 원하나(성공 기준) ⑤ 증거는(로그/VOC/통계 등 근거)
- **Step 3 — 문제 정의서 표**: AS-IS / 발생 빈도 / 구체적 피해 / TO-BE / 측정 가능한 성공 기준 / 증거 소스 / REQ ID — 7항목. 사용자가 "맞아요" 하기 전까지 진행 금지
- **Step 4 — PRD 진입 전 승인 체크 4항목**: 문제 진술 한 문장 요약 가능 / 피해가 최소 1개 수치로 표현 / TO-BE가 검증 가능한 사용자 행동 / 근거 소스 1개 이상
- 말미에 AICC(AI 컨택센터) 도메인 특화 "표면적 요청 → 진짜 문제" 매핑 표 포함

### 02. 사용자 관점 전환 (`02_user_perspective.md`)

기획자 본인이 아니라 **실제 사용자 맥락**에서 모든 결정을 검증하는 스킬.

- **사용자 파악 질문 7개** (기획 시작 전 필수): 연령대, 사용 시점·장소, 시스템 숙련도, 사용 직전/직후 행동, 가장 두려운 실패 시나리오, 현재 대체 중인 과정 등
- **추상어 금지 번역표** (핵심 규칙): "간편하게"→"3번 탭 이내로", "빠르게"→"2초 이내에", "쉽게 찾을 수 있는"→"메인 화면에서 1단계 접근". "편리한 UI"/"직관적인" 등은 아예 금지
- **페르소나 2명을 판단 기준으로 강제**:
  - 상담사 정해린(29세, 옴니채널, 핵심 사용자) — 동시 응대·단축키 위주. 불안: "이전 상담 내역을 못 찾을까 봐"
  - 상담관리자 한승우(41세, 12명 관리, QA·운영) — 불안: "오분류로 지표 왜곡"
- **페르소나 검증 질문**: 기능마다 "정해린이 동시 응대 중 3초 안에 파악 가능한가?" + "한승우가 1단계로 드릴다운 가능한가?" — **둘 다 Yes가 아니면 설계 수정**
- 결정 근거는 ADR 형식(결정/이유/버려진 대안과 거절 이유)으로 기록

### 03. 구조화 능력 (`03_structuring.md`)

요구사항을 계층으로 정리하고 의존관계를 파악해 MVP 경계를 확정하는 스킬.

- **REQ ID 체계**: `REQ-001` 형식, PRD/IA/플로우/화면/테스트 전체에서 동일 ID 사용
- **기능 트리 5계층**: 서비스 → 도메인 → 기능 그룹 → 세부 기능 → 엣지케이스, 각각 REQ 연결
- **우선순위 3단계**: `MUST / SHOULD / NICE` — **MUST 비율이 40%를 넘으면 과설계로 판단하고 조정**
- **MVP 경계 기준**: 포함 = 핵심 가치 플로우를 막는 기능·법적/보안 필수 / 제외 = 편의 기능·개인화·통계 대시보드·발생 빈도 5% 미만 엣지케이스
- 산출물은 도메인별 "MVP(v1) / v2 예정 / 보류·미포함(+ADR 근거)" 3구획으로 정리. 각 기능에 의존 관계·엣지케이스 1개 이상·검증 시나리오·오너 필수
- AICC 기능 도메인 맵 6개(상담 인입/처리/고객·이력/운영/분석/품질 QA) 포함 — 기능 요청이 오면 지도에서 위치부터 확인

### 04. 엣지케이스 감지 (`04_edge_cases.md`)

정상 플로우의 구멍을 개발 전에 찾아내는 스킬 (개발 후 발견 시 비용 10배, 서비스 중 발견 시 100배라는 전제).

- **우선순위 분류 축**: 영향도 × 빈도 × 대응 난이도 → MUST/SHOULD/NICE
- **범용 체크리스트 6개 카테고리 20개 질문** (모든 기능에 적용 필수):
  - 동시성(3) — 동일 자원 동시 점유, 저장 2회 연타, 설정 수정과 사용의 동시 발생
  - 네트워크/시스템(3) — 저장 직전 종료, 전송 중 서버 다운, 연동 단절
  - 데이터 경계값(3) — 특수문자·이모지, 잘못된 형식, 글자 수 초과
  - 시간/날짜(4) — 자정 넘는 집계, 영업시간 외, SLA 직전, 종료 건 재오픈
  - 상태 변화(4) — 담당자 이탈, 사용 중 코드 삭제, 채널 전환, 규칙 변경
  - 권한/접근(3) — 세션 만료 중 저장, 타 팀 조회, 탈퇴 데이터 처리
- **EC 문서 포맷**: `EC-[번호]: [상황명] (REQ-###)` 헤딩 + 7행 표(발생 조건/빈도/영향/처리 방법[무시·안내·자동 복구·강제 중단 중 택1]/우선순위/처리 화면/MVP 포함 여부)
- **게이트**: MUST 엣지케이스 하나라도 누락 시 UI 문서 단계 진입 금지

### 05. 문서 작성 노하우 (`05_document_writing.md`)

읽는 사람이 질문 없이 이해하고 바로 행동할 수 있는 문서를 쓰는 스킬.

- **기본 원칙 5개**: 결론 우선(피라미드), 한 문장 한 의미, 추상어 삭제→수치 대체, ADR 기록, ID 중심 구조
- **독자별 전략**: 개발자(조건·예외·데이터 형식, 상세) / 대표·클라이언트(임팩트·수치·일정, 1페이지) / 디자이너(감정·플로우·상태 중심)
- **PRD 필수 템플릿 15개 섹션** (이 파일의 핵심): 문서 정보(버전·상태) → 한줄 요약 → 문제 정의(AS-IS/TO-BE/성공 기준/증거) → **요구사항 레지스트리 표** → 스코프 → 사용자 스토리 → 기능 상세 → NFR → 결정 로그(ADR) → 가정✅/확정🔲/미결❓ → 테스트 케이스 초안 → 변경 이력 → 관련 문서 추적
- **Mermaid 플로우차트 규칙**: happy path 먼저, 분기점마다 조건 명시, 오류 플로우 별도, 단계마다 액터 명시
- **발행 전 자가 검토 7문항** — 하나라도 No면 즉시 수정 (개발자가 질문 없이 구현 가능한가, 대표가 30초에 파악 가능한가 등)
- **변경 기록(CR) 규칙**: v1.0 확정 이후의 모든 변경은 `prd/CHANGELOG.md`에 CR-### 레코드로 기록 — AS-IS/TO-BE 문구 그대로 인용, 방향성 결정은 ADR과 상호 링크(ADR=의사결정 기록, CR=실행 기록). 변경 범위는 3단계로: ① `impact_scan.py [대상ID] --md`로 확인 대상 파일 체크리스트 자동 생성 → ② 파일별 확인·수정 기록 → ③ `validate_traceability.py`로 ID 누락 재확인. 자연어 변경 요청 접수부터의 전체 절차는 스킬 14 참조

### 06. 테스트 케이스 도출 (`06_test_cases.md`)

확정된 REQ·EC를 검증 가능한 형태(Given-When-Then)로 **번역**하는 스킬. 04 완료 후, 05로 PRD 조립하기 **직전**에 수행합니다.

- **핵심 원칙**: ① TC는 창작이 아니라 번역 ② Then은 반드시 관찰 가능·수치화("잘 동작한다" ❌ → "3초 이내 응답 시작" ✅) ③ **TC 헤딩 괄호에 검증 대상 REQ/EC 명시** — `### TC-001 (REQ-001, EC-03)` — 추적성 스크립트가 이 괄호를 읽어 커버리지를 자동 계산 ④ 우선순위는 검증 대상 REQ에서 상속
- **커버리지 목표**: 모든 MUST REQ → 정상 TC 최소 1개 / MUST·SHOULD EC → 예외 TC 최소 1개 / 측정값 있는 NFR → 성능 TC 최소 1개
- 예외 TC는 EC의 "발생 조건"=Given, "처리 방법"=Then으로 그대로 옮김
- `validate_traceability.py --strict`에서 "TC 없는 MUST REQ" ❌가 0이 될 때까지 반복

## Phase 2: UI 문서 (07~10)

### 07. 서비스 구조도 / IA (`07_information_architecture.md`)

PRD의 기능(REQ)을 **화면 구조**로 변환해 화면 누락을 방지하는 스킬.

- 화면 추출 규칙: URL 하나 = 1화면, 모달/바텀시트 = 하위 상태, 사용자 유형이 다르면 별도 루트
- **각 화면에 REQ 최소 1개 연결 의무**, 화면당 핵심 액션 최대 5개, 메뉴 **깊이 3단계 초과 금지**
- 산출물: 트리 구조(`화면명 (SC-##) - 설명 - REQ-###`) + 공통 화면(로그인·오류·빈 상태) + 매핑표(`SC-ID | 화면명 | REQ-ID | 우선순위`)
- "누락된 화면이 있나요?" 사용자 확인 전까지 다음 단계 진행 금지

### 08. 사용자 플로우 (`08_user_flow.md`)

목표 달성까지의 화면 이동 경로를 정의하는 스킬 — 화면 설계와 테스트의 기초.

- 모든 플로우는 3요소 필수: **정상 경로 / 분기 조건 / 예외 경로**(실패·취소·권한 이슈 후 복귀 지점 포함)
- 반드시 Mermaid flowchart 사용 — 사각형=화면, 마름모=분기, 원형=시작/종료, 화살표 라벨=액션/조건
- **승인 전 게이트**: 모든 핵심 액션에 목적지/결과 존재, 예외 경로에 복귀 지점 명시, **플로우당 최소 1개 에러 메시지 문안 연결**
- 완성 기준: IA의 모든 화면이 최소 1개 플로우에 등장 + Given/When/Then 검증 시나리오 첨부

### 09. 화면 명세서 (`09_screen_spec.md`)

"UI 구현자가 추측 없이 그릴 수 있는 설계도"를 만드는 스킬.

- **전체 일괄 생성 금지** — 화면 하나씩 순차 생성 (플로우 등장 순 → 기능 밀집도 → 예외 많은 순)
- **화면당 필수 5항목**: ① 표시 데이터(항목 단위 구체 작성 — "문의 정보" ❌ → "인입 시각, 채널, 고객명, 문의 유형" ✅) ② 사용자 액션(목적지/결과/제약) ③ 상태(로딩/빈값/오류/성공) ④ 입력 규칙(유효성 + **실제 에러 메시지 텍스트**) ⑤ 연결 정보(REQ/SC/TC)
- 성공 플로우만 쓰지 말 것: 네트워크 실패·동시성 충돌·세션 만료 처리까지 미세 명세
- 레이아웃 힌트(상단/중간/하단)를 남겨 Phase 3 와이어프레임의 기준으로 사용

### 10. 디자인 방향 (`10_design_direction.md`)

기획된 플로우·화면을 일관된 시각 체계로 바인딩하는 스킬.

- **질문 없이 디자인 방향 임의 결정 금지** — 필수 질문 4개: ① 참고 서비스 ② 한 단어 느낌(톤) ③ 주요 사용 환경(기기) ④ 선호·회피 색상
- 시스템 값 5가지 정의: 컬러 팔레트(Primary/Secondary/Neutral/Danger) / 타이포그래피 / 레이아웃 기준 / 컴포넌트 스타일 / 톤 & 매너
- 대안 1개 이상을 ADR로 기록: 채택 이유 / 거절 이유 / 어떤 페르소나에 유리한지
- AICC 기본 팔레트 참고 예시 포함 (Primary #2563EB 딥 블루 등, 데스크톱 1280px 우선)

## Phase 3: UI 생성 (11 + visual_generation)

### 11. UI 생성 (`11_ui_generation.md`)

화면 명세서 + 디자인 방향을 바탕으로 실제 동작하는 샘플 HTML을 생성하는 스킬.

- **시작 전 필수**: `ui/` 4개 문서 모두 읽기 — 하나라도 없으면 생성 시작 금지
- **화면당 5단계 절차** (본 HTML 바로 생성 금지, 각 단계마다 사용자 확인):

  ```
  ① Mermaid 구조도            → 확인
  ② 저해상도 HTML 와이어프레임  → 확인   (그레이스케일, Type B)
  ③ 주석 HTML 도해(콜아웃)     → 확인   (동작 규칙·조건·분기 설명, Type C)
  ④ 본 HTML + Tailwind CSS    → 확인   (디자인 반영, 상태 구현)
  ⑤ 다음 화면으로
  ```

- **구현 강제 항목**: 동작하는 링크 이동, 실제 유효성 메시지 텍스트, 빈/로딩/에러 상태 최소 1개씩, 실패 시 복구 버튼(재시도/뒤로가기)
- **더미 데이터 고정 명단**: 고객 김지수·박서연·이민수·최유리·정하준 / 상담사 정해린·박도윤·김서아 / 관리자 한승우 — **영문 사용 금지**
- 각 화면 파일 상단에 REQ/SC/TC 추적 주석 명시, 화면 이동은 파일 간 링크(`location.href`)로 실제 클릭 가능하게
- 검수: TC 기준 핵심 시나리오 1개 이상을 화면에서 수동 실행 가능해야 함

### visual_generation (`SKILL.md` + `REFERENCE.md`)

시각 자료를 외부 이미지 생성 API 없이 HTML/Mermaid로 직접 만드는 스킬. 15개 번호 스킬과 달리 프론트매터가 수작업 커스텀(`disable-model-invocation: true` — 명시 호출 전용)이라 동기화 대상에서 제외됩니다.

- **3가지 생성 유형**:
  | 유형 | 산출물 | 도구 | 용도 |
  |---|---|---|---|
  | Type A | 서비스 플로우 다이어그램 | Mermaid (문서 인라인) | PRD/플로우 보완, 노드 최대 10개 |
  | Type B | UI 컨셉 와이어프레임 | 그레이스케일 HTML | 레이아웃 확인 — 일부러 "미완성 컨셉" 룩 유지 |
  | Type C | 주석 HTML 도해 | HTML + 절대위치 콜아웃 | 동작 규칙·조건·분기를 번호 뱃지(①②③)로 설명 |
- **콜아웃 색상 규약 5색**: 빨강=오류·제약·비활성 / 파랑=사용자 액션·이동 / 초록=활성·완료·정상 / 주황=경고·트리거 / 회색=기본 동작·닫힘
- 저장 규칙: `docs/[프로젝트명]/assets/wireframes/wf-[SC-ID]-[상태명].html`, `annotated-[SC-ID]-[라벨].html` — 해당 문서에 상대경로 링크 연결
- 콜아웃·라벨 반드시 한국어, Type C 콜아웃 텍스트는 기획서 동작 규칙 문구와 일치시킴

## Phase 4: 개발 핸드오프 (12~13)

### 12. 기능명세서 (`12_functional_spec.md`)

PRD(서술)와 화면 명세(레이아웃) 사이의 빈칸인 **동작 규칙**을 `FS-###` 단위로 정의하는 개발·QA 핸드오프 문서. Phase 3(UI 생성)과 병행 가능.

- **핵심 원칙 6개**:
  1. FS는 창작이 아니라 **분해** — REQ 1개→FS 여러 개, PRD에 없는 기능을 FS에서 새로 만들지 않음(스코프 크립 방지)
  2. 모든 FS는 **입력→처리 규칙→출력→예외** 4칸 완결 — 하나라도 못 채우면 미결(❓)
  3. 모든 FS에 REQ/SC/TC 연결 명시
  4. 조건 3개 이상 얽힌 분기는 별도 **결정표(D-#)**로 분리
  5. 전 기능 공통 규칙(인증·마스킹·폴백 등)은 공통 규칙 섹션으로 승격 (같은 문장 3번째 반복 시)
  6. **기술 구현 방법 기재 금지** — "무엇이 일어나야 하는가"까지만
- 실행 7단계: 범위·독자 확인 질문 → REQ 체크리스트화(MUST 커버리지 100%) → REQ→FS 분해 → 공통 규칙 분리 → 결정표 분리 → 추적 연결(우선순위는 REQ에서 상속, 임의 조정 금지) → 역추적표 + `--strict` 검증
- 산출물: `fnspec/기능명세서.md` — 기능 명세 표(`FS-ID|기능명|처리 규칙|입력|출력|예외(EC)|우선순위|연결`) + 결정표 + 역추적표

### 13. WBS 작성 (`13_wbs.md`)

확정된 스코프(REQ·FS)를 **일정과 리소스**로 변환하는 스킬. 기능명세서 없이 작성 금지.

- **핵심 원칙 5개**:
  1. **명목일이 아닌 가용일로 계획** — 실제 달력(주말·법정공휴일·대체공휴일) 대조 필수
  2. **작업 단위 발명 금지** — Epic=REQ, Story=FS. FS에 없는 작업(인프라 등)은 별도 섹션+근거
  3. **외부 의존(전제)은 지뢰** — 🔴 표기 + "미제공 시 어느 주차로 후퇴하는가" 사전 정의
  4. **대외 약속 불변, 내부 목표 선행** — 고객사 게이트는 고정, 내부 목표를 1~2주 앞당겨 버퍼 확보
  5. 임의 산정값은 "(가정, 착수 시 확정)" 명시
- 두 개의 판: **기능판**(FS를 몇 주차에, 티켓 생성 기준 — 의존 FS가 선행 FS보다 앞 주차 배치 금지) + **리소스판**(멤버×주차 투입률 %, 합계 100% 초과 = 배치 오류)
- 게이트 통과 기준은 **수치화 필수** ("검수" ❌ → "Intent 정확도 75%·SOP 사인오프" ✅)
- 가용일 3~4일인 주는 ⚠️ 표시 + 용량 축소, 버퍼 주에는 기능 배치 금지

## 운영 단계: 변경 관리 (14)

### 14. 변경 요청 처리 (`14_change_request.md`)

v1.0 확정 이후 자연어로 들어온 요청("보험료 계산에 납입기간도 들어가야 해요")을 안전하게 반영하는 스킬. 특정 Phase에 속하지 않고 문서 확정 이후 **운영(변경 관리) 국면**에서 사용됩니다.

- **공통 접수(Step 0)**: 요청이 어느 REQ인지 레지스트리에서 후보를 찾아 **사용자에게 판별 확정** — 후보가 1개여도 반드시 확인, 애매하면 "기존 REQ 확장 / 신규 추가" 양쪽을 선택지로 제시. 판별 결과에 따라 두 플로우로 갈라집니다:
- **A플로우 — 기존 요구사항 변경**:
  1. 문서를 고치기 **전에** `impact_scan.py`로 영향 범위 스캔 → "문서 N개 M곳에 영향" 보고
  2. AS-IS(현재 문구 그대로 인용) / TO-BE 변경안 제시 → 사용자 승인 (방향성 결정은 ADR 먼저)
  3. **PRD부터** 수정 → 연쇄 문서를 **한 파일씩 컨펌받으며** 수정 — 파일마다 "수정 필요(이렇게) / 영향 없음(사유)" 판단을 보여주고 승인 후 진행, 일괄 수정 금지
- **삭제는 A플로우의 변형** — "TO-BE가 폐기인 변경"으로 취급해 같은 골격으로 처리하되, 삭제 특유 규칙 4가지를 지킵니다:
  - 레지스트리 **행 물리 삭제 금지** — 우선순위 칸만 `폐기(CR-###)`로 바꿈. 행을 지우면 CHANGELOG·ADR의 과거 언급이 전부 유령 참조 ❌가 되어 검증이 영원히 실패하기 때문. 폐기된 REQ는 MUST/SHOULD/NICE 어느 것도 아니게 되어 TC 커버리지·고아 검사에서 자동 제외됨(스크립트 수정 불필요)
  - 판별 시 삭제 유형 확정: **완전 폐기 / 다음 버전으로 연기 / 다른 REQ로 흡수** (흡수면 그 REQ의 일반 변경으로 전환)
  - 연쇄 문서 판단이 둘로 갈림: **활성 문서**(ui·fnspec·wbs)는 참조 제거, **기록 문서**(CHANGELOG·ADR)는 보존. 이 REQ만 검증하던 TC·전용 화면 등은 동반 폐기 대상으로 컨펌
  - 폐기 REQ는 정의가 남아 있어 잔존 참조를 스크립트가 못 잡으므로 **impact_scan 체크리스트 완주가 곧 게이트**
- **B플로우 — 요건 추가 (신규 REQ)**: 새 ID는 아직 어디에도 없어 impact_scan을 쓸 수 없으므로 접근이 다릅니다:
  1. **스코프 판정 먼저** — ID 채번(결번 재사용 금지), 우선순위(MUST 40% 초과 시 과설계 경고), v1.x 포함 vs v2 백로그(WBS 있으면 일정 영향 보고) → 사용자 결정 + ADR 기록
  2. **완전한 정의 작성** — 스킬 01·04·06 축약 적용: 문제 근거 수치 1개, EC 최소 1개, MUST면 TC 필수
  3. **산출물 단계 체크리스트로 전파** — IA(기존 화면에 얹나/신규 SC 채번하나) → 플로우 → 화면 명세 → FS 분해 → WBS 배치, 파일 단위 컨펌은 A와 동일
  4. `--strict` 검증이 누락 적발의 주역 — 새 MUST REQ 등록 순간 "TC 없는 MUST ❌", "화면에 없는 MUST ❌"가 빠진 전파를 자동으로 잡음
- **공통 마감**: `validate_traceability.py --strict` ❌ 0 + CR-### 레코드 기록 (추가의 경우 AS-IS 칸은 "없음(신규 추가)")
- CR 레코드 형식은 스킬 05의 "변경 기록(CR) 작성 규칙"을 따름

### 15. 내용 정합성 검증 (`15_consistency_check.md`)

추적성 검증(ID 연결)과 기계 검사가 못 잡는 **문장 의미 충돌**(PRD "3초" vs FS "5초" 같은)을 찾아내는 스킬. Phase 게이트 직전·대규모 변경 후·사용자 요청 시 사용합니다.

- **3계층 구조**: ① `validate_traceability.py --strict`(ID 연결) → ② `consistency_check.py --strict`(기계 판정: 우선순위 상속·단계 일치·MUST 배치·산출물 존재·폐기 참조·FS 중복 ID) → ③ 의미 대조(본 스킬)
- **의미 대조 절차**: 4영역(PRD↔FS / PRD↔UI / 명세↔시각물 / FS↔WBS·SOW) 병렬 에이전트 검사 → 발견을 **[결정]/[기계적]/[상류 CR]**로 분류 → [결정]은 건별 `AskUserQuestion`으로 사용자 확인(임의 판정 금지), 판정 기준은 문서 서열(PRD 상류 우선) → 파일 영역별 병렬 수정 → 스크립트 재검증으로 마감
- **금지**: 스크립트 검증 건너뛰고 의미 대조부터 시작, [결정] 항목 임의 판정, 표현·상세도 차이를 충돌로 보고
