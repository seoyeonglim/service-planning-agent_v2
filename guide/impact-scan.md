# 변경 영향 스캔의 동작 원리 (impact_scan.py)

> 이 문서는 [README](../README.md)의 상세 가이드입니다.

`impact_scan.py`는 한 마디로 **"REQ-041을 바꿀 건데, 어느 문서들을 같이 열어봐야 하지?"에 답해주는 수색기**입니다.

요구사항 하나는 PRD에만 있지 않습니다. 화면 명세서, 사용자 플로우, 테스트 케이스, 기능명세서, 심지어 HTML 화면 상단 주석까지 곳곳에 흩어져 언급됩니다. 그래서 v1.0 확정 후 요구사항을 하나 고치면 그 ID가 등장하는 모든 문서를 찾아 함께 손봐야 하는데, 사람이 기억에 의존해 찾으면 반드시 한두 개를 빠뜨립니다. 그 수색 작업을 기계가 대신해주는 것입니다.

## 실행 방법

```bash
python3 .claude/scripts/impact_scan.py REQ-041                  # REQ-041이 등장하는 파일 + 줄 번호 출력
python3 .claude/scripts/impact_scan.py REQ-041 EC-03 TC-018     # 여러 ID를 한 번에 스캔
python3 .claude/scripts/impact_scan.py REQ-041 --project 이름    # 특정 프로젝트 폴더만 스캔
python3 .claude/scripts/impact_scan.py REQ-041 --md             # CHANGELOG에 붙여넣을 체크리스트 형태로 출력
```

기본 출력은 이런 모양입니다 — 파일마다 몇 번째 줄에 등장하는지까지 알려주므로 바로 찾아갈 수 있습니다:

```
🗂  [프로젝트명] — 대상: REQ-041
  □ prd/PRD.md  (6곳: L88, L214, L390, ...)
  □ ui/03_screen_spec.md  (2곳: L45, L112)
  □ ui/screens/sc-05-detail.html  (1곳: L3)
  └─ 총 3개 파일 — 각 파일을 확인하고 CR의 `변경 범위` 칸에 결과를 기록하세요.
```

`--md`를 붙이면 같은 내용이 `- [ ] prd/PRD.md (6곳)` 형태의 마크다운 체크리스트로 나옵니다. 이것을 CR(변경 기록)의 "변경 범위" 칸에 초안으로 붙여넣고, 파일을 하나씩 확인할 때마다 체크하는 용도입니다.

## 알아두면 좋은 것들

- **다섯 종류 ID 모두 스캔 가능**: `REQ-###`(요구사항) / `EC-##`(엣지케이스) / `TC-###`(테스트) / `SC-##`(화면) / `FS-###`(기능명세)
- **축약 표기도 놓치지 않음**: 문서에 `REQ-001~005`(범위)나 `REQ-070·071`(나열)처럼 축약해 적어도 개별 ID로 풀어서 인식합니다 — `validate_traceability.py`와 같은 인식 로직을 공유하므로 두 스크립트의 판단이 어긋나지 않습니다
- **변경 대상이 아닌 곳은 자동 제외**: `_archive/`(과거 버전 보관함), `ref/`(고객사가 준 원본 자료), `_traceability_report.md·_consistency_report.md`(자동 생성물)는 스캔하지 않습니다
- **오타 감지**: 지정한 ID가 어디에도 없으면 "ID 오타이거나 아직 정의 전일 수 있습니다" 경고와 함께 실패(종료코드 1)로 끝납니다 — 잘못된 ID로 헛수고하는 것을 초입에서 막아줍니다

## 주 사용 시점 — 변경 기록(CR) 작성 3단계

v1.0 확정 이후 변경이 생기면 이 순서로 사용합니다:

1. `impact_scan.py [대상ID] --md` → 확인 대상 파일 체크리스트 자동 생성
2. 체크리스트의 파일을 하나씩 열어 확인·수정하고 결과를 CR의 "변경 범위" 칸에 기록
3. `validate_traceability.py --strict` → 수정 과정에서 ID가 깨지거나 누락되지 않았는지 재확인

자연어 변경 요청 접수("~바꿔주세요")부터 REQ ID 매핑 확정, 파일별 컨펌 수정까지의 **전체 플로우는 스킬 14(변경 요청 처리)**가 담당하며, 위 3단계는 그 안에 포함되는 실행 단계입니다. → [스킬 상세: 14. 변경 요청 처리](skills.md#14-변경-요청-처리-14_change_requestmd)

> ⚠️ **한계를 알고 쓰세요**: 이 스크립트는 "어디를 봐야 하는지"까지만 알려줍니다. 각 문서의 내용이 변경된 요구사항과 실제로 일치하게 고쳐졌는지는 사람이(또는 Phase 게이트에서) 직접 대조해야 하며, 내용이 서로 어긋날 때의 판정은 문서 서열 규칙(CLAUDE.md 공통 운영 규칙: PRD > UI 문서 > 기능명세서·화면 HTML > WBS, 상류 우선·하류 수정)을 따릅니다. 내용 검증 전체 체계는 `consistency_check.py`(기계 판정) + 스킬 15(의미 대조)의 3계층 검증 참조.
